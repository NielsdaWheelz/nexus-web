# Evidence-Grade Citations Hard Cutover

## Purpose

Replace message-level source chips with production-grade claim/evidence
citations.

The final product model is:

- every source-grounded assistant answer has a durable evidence summary,
- every checkable assistant claim has an explicit support state,
- every supporting citation is a backend-owned evidence object,
- every evidence object includes an exact snippet and jump target when the source
  has an in-app location,
- scoped chats say when the current source scope does not contain enough
  evidence.

This is a hard cutover. The final state has no source-chip citation fallback, no
model-authored citation ids, no legacy bracket-marker parsing, no compatibility
mode for old citation shapes, no prompt-only citation guarantee, and no hidden
unscoped retrieval expansion when scoped evidence is insufficient.

## Goals

- Make citations evidence-grade: exact snippet, source location, retrieval
  status, support state, and jump target.
- Align citations to answer claims, not only to whole messages.
- Keep citation authority in persisted backend objects.
- Make insufficient evidence a first-class answer state.
- Preserve durable chat runs, tool calls, retrieval rows, prompt assembly, and
  reader deep links as the foundation.
- Normalize app, web, attached-context, and memory evidence behind one source
  reference contract.
- Record whether evidence was retrieved, selected, and included in the prompt.
- Verify every persisted claim/evidence link before rendering it as supported.
- Make citation coverage observable for operators and testable in CI.
- Keep public-web citations visually and semantically distinct from app-source
  evidence.
- Keep the BFF transport-only and domain logic in Nexus backend services.

## Target Behavior

### Assistant Answers

- Assistant answers render as normal prose with inline claim markers only where
  a claim has evidence metadata.
- A claim marker opens the claim evidence view for that exact claim.
- The evidence view shows:
  - claim text,
  - support state,
  - exact snippet,
  - source title,
  - page, section, fragment offset, timestamp, URL, or message location,
  - retrieval status,
  - selected status,
  - prompt inclusion status,
  - score when available,
  - jump target.
- Evidence snippets are copied from source snapshots or retrieved chunks. They
  are not regenerated by the model.
- App-source evidence jumps into the reader, transcript, conversation, or search
  result location.
- Web evidence opens the cited URL and displays the web result title, URL,
  provider, snippet, and access time.
- Claims without enough support are rendered with a non-support state, not a
  source citation.
- A completed answer never displays a claim as supported while claim alignment or
  verification is still pending.

### Scoped Chats

- Media-scoped answers can cite only evidence from the media scope, attached
  per-message contexts, memory refs that resolve to authorized evidence, or an
  explicit web-search tool result.
- Library-scoped answers can cite only evidence from the library scope, attached
  per-message contexts, memory refs that resolve to authorized evidence, or an
  explicit web-search tool result.
- Scoped app retrieval never falls back to all-library or unscoped retrieval.
- If scoped retrieval finds no useful evidence for a source-grounded question,
  the answer begins with an explicit insufficient-evidence state.
- The UI shows the insufficient-evidence state with the scoped corpus name and
  retrieval status.
- General guidance may follow only after the insufficient-evidence state is
  rendered and persisted.

### Retrieval Status

Each candidate evidence item carries one retrieval status:

- `attached_context`: user attached this source to the turn.
- `retrieved`: the tool returned this source.
- `selected`: the retrieval pipeline selected this source for answer context.
- `included_in_prompt`: the prompt assembly included this source text.
- `excluded_by_budget`: the source was retrieved but omitted by budget.
- `excluded_by_scope`: the source was rejected by scope or permission policy.
- `web_result`: the source came from explicit web search.

`included_in_prompt` is the only app-retrieval status that can support a
source-grounded claim by default. A verifier may use retrieved-but-not-included
items only for negative audit states such as contradiction or insufficient
evidence.

### Support States

Each claim has one final support state:

- `supported`: evidence entails the claim.
- `partially_supported`: evidence supports part of the claim but not all
  material details.
- `contradicted`: evidence conflicts with the claim.
- `not_enough_evidence`: in-scope evidence is missing or too weak.
- `out_of_scope`: the answer would require sources outside the current scope.
- `not_source_grounded`: the claim is conversational, procedural, or reasoning
  glue and does not require source evidence.

Only `supported` claims render as positive citations. Other states render as
explicit evidence diagnostics.

### Streaming

- Answer text may stream before evidence verification completes.
- During streaming, claim markers may be absent or pending.
- A message is not final until claim extraction, evidence alignment, and
  verification have either completed or failed with a persisted terminal error.
- If evidence finalization fails, the message is marked incomplete and no legacy
  source chips are rendered.

## Final State

### Kept

- `POST /api/chat-runs` as the only send endpoint.
- Durable chat runs as the execution source of truth.
- `message_tool_calls` as durable tool-call records.
- `message_retrievals` as durable retrieval records.
- `chat_prompt_assemblies` as durable prompt inclusion ledgers.
- `content_chunks` as the semantic retrieval index for text-bearing media.
- `ConversationSourceRef` as the typed frontend source-ref concept.
- Reader deep links for EPUB fragments, PDF page geometry, transcript
  timestamps, and conversation locations.
- Existing web-search tool separation from app retrieval.

### Removed

- Message-level source chips as the citation UI.
- `SourceCitations` as the final assistant citation renderer.
- Any UI path that treats `result_ref.title` as sufficient citation evidence.
- Any final answer citation that depends on model-authored bracket markers.
- Any compatibility renderer for old web citation chip arrays.
- Any citation event that is final without persisted evidence rows.
- Any prompt instruction that is the only guard against citation hallucination.
- Any scoped-chat behavior that expands retrieval after insufficient evidence.
- Any client-provided quote preview used as evidence text.
- Any citation jump target built ad hoc in the frontend.

## Architecture

```text
ChatSurface
  MessageRow
    ClaimEvidenceMessage
      answer text with claim markers
      claim evidence panel
      insufficient evidence state

Next.js BFF
  transport-only /api/* proxy

FastAPI routes
  validate request
  call services
  return persisted evidence summaries with messages
  stream evidence status events

Services
  chat_runs
    execute retrieval tools
    assemble prompt
    stream assistant answer
    finalize claim/evidence state

  agent_tools.app_search / web_search
    return normalized evidence candidates
    persist retrieval rows with locators and snippets

  context_assembler
    records selected and included evidence ids
    hydrates attached context and memory source refs

  evidence
    normalizes source refs and locators
    snapshots exact snippets
    builds jump targets

  claim_alignment
    extracts atomic answer claims
    proposes evidence links from included sources
    persists claim rows

  evidence_verifier
    validates source permission, scope, snippet, locator, and support state
    rejects hallucinated or out-of-scope citations
```

The frontend renders only persisted evidence summaries and claim/evidence rows.
The model may propose claim/evidence alignment, but the verifier owns the final
persisted state.

## Data Model

### Message Retrievals

Extend retrieval persistence so each row can act as a candidate evidence
snapshot.

Required durable fields:

- source type,
- source id,
- media id when applicable,
- conversation scope at retrieval time,
- result ref,
- context ref,
- source title,
- section label,
- exact snippet,
- snippet prefix,
- snippet suffix,
- locator metadata,
- deep link,
- score,
- retrieval status,
- selected flag,
- included-in-prompt flag,
- source version,
- created_at.

Rules:

- Retrieval locators use typed JSON, not stringified JSON.
- Retrieval snippets are source snapshots. They survive source reindexing.
- All offsets are right-open `[start, end)` intervals.
- Permission failures are not persisted as evidence candidates.
- Scope rejection is persisted only as an audit status without snippet text.

### Evidence Summary

Add one evidence summary row per assistant message.

Required fields:

- `id`
- `message_id`
- `scope_type`
- `scope_ref`
- `retrieval_status`
- `support_status`
- `verifier_status`
- `claim_count`
- `supported_claim_count`
- `unsupported_claim_count`
- `not_enough_evidence_count`
- `prompt_assembly_id`
- `created_at`
- `updated_at`

Rules:

- A completed assistant message has exactly one evidence summary.
- `verifier_status=pending` is allowed only while the chat run is active.
- Final summaries are immutable except for operator-visible invalidation.

### Claims

Add `assistant_message_claims`.

Required fields:

- `id`
- `message_id`
- `ordinal`
- `claim_text`
- `answer_start_offset`
- `answer_end_offset`
- `claim_kind`
- `support_status`
- `verifier_status`
- `created_at`

Rules:

- Claim spans point into the final assistant message text.
- Claim spans are optional only for whole-message insufficient-evidence states.
- Claim text is copied from the final answer or a deterministic normalized span.
- Claims are atomic enough that one material proposition can be verified.
- Claims that do not need source grounding use `not_source_grounded`.

### Claim Evidence

Add `assistant_message_claim_evidence`.

Required fields:

- `id`
- `claim_id`
- `ordinal`
- `evidence_role`
- `source_ref`
- `retrieval_id`
- `context_ref`
- `result_ref`
- `exact_snippet`
- `snippet_prefix`
- `snippet_suffix`
- `locator`
- `deep_link`
- `score`
- `retrieval_status`
- `selected`
- `included_in_prompt`
- `source_version`
- `created_at`

Evidence roles:

- `supports`
- `contradicts`
- `context`
- `scope_boundary`

Rules:

- `source_ref` is the canonical typed evidence identity.
- `retrieval_id` is required for retrieved app and web evidence.
- Attached message contexts may omit `retrieval_id` and must include
  `context_ref`.
- `exact_snippet` is required for `supports` and `contradicts`.
- A claim can have multiple supporting evidence rows.
- Evidence rows snapshot display data; the UI does not rehydrate snippets to
  render a completed message.

- `epub_fragment_offsets`
  - `media_id`
  - `section_id`
  - `fragment_id`
  - `start_offset`
  - `end_offset`
- `pdf_page_geometry`
  - `media_id`
  - `page_number`
  - `quads`
  - `exact`
  - `prefix`
  - `suffix`
- `transcript_time_range`
  - `media_id`
  - `transcript_version_id`
  - `t_start_ms`
  - `t_end_ms`
- `conversation_message`
  - `conversation_id`
  - `message_id`
  - `message_seq`
- `web_url`
  - `url`
  - `title`
  - `display_url`
  - `accessed_at`
- `external_source`
  - `source_name`
  - `source_id`
  - `url`

Rules:

- App evidence with a locator must have a deep link.
- EPUB and PDF locators must be precise enough to focus the cited span.
- Web locators never claim page offsets unless the web-search provider supplied
  verified offsets.
- Locator parsing happens at the backend boundary.

## Prompt And Generation Contract

- Prompt context is rendered from backend-owned evidence candidates.
- Each included evidence block has a stable evidence id that maps to a persisted
  retrieval or attached context.
- Evidence block ids are opaque. The user never sees raw ids.
- The assistant may use evidence ids only in the structured sidecar output.
- The final user-visible answer does not contain citation bracket markers.
- The model is instructed to separate answer text from claim/evidence proposals.
- Claim/evidence proposals are treated as untrusted until verified.
- The backend removes, rewrites, or marks unsupported claim proposals before
  finalizing the message.
- Prompt XML values are escaped inline where generated text is built.

## Pipeline

For every source-grounded assistant turn:

1. Load conversation scope, message contexts, and authorized memory source refs.
2. Build the retrieval plan.
3. Execute app retrieval and explicit web search when requested.
4. Persist every retrieval candidate with normalized locator and snippet data.
5. Assemble prompt context and persist the prompt assembly ledger.
6. Generate answer text and a structured claim/evidence proposal.
7. Extract or normalize atomic claims from the final answer.
8. Align claims to included evidence candidates.
9. Verify scope, permission, snippet, locator, and support state.
10. Persist evidence summary, claims, and claim evidence rows.
11. Stream evidence status events.
12. Render the assistant message from persisted evidence rows.

If step 9 cannot verify a source-grounded claim, the claim is not rendered as
supported. If the answer depends on that unsupported claim, the final answer
must be repaired or marked incomplete.

## Rules

- Hard cutover only.
- No feature flag.
- No legacy source-chip fallback.
- No model-authored visible citations.
- No bracket-marker parsing for final citations.
- No old web citation chip compatibility path.
- No scoped retrieval fallback to unscoped sources.
- No client-provided quote preview as evidence.
- No evidence row without source permission at write time.
- No supported claim without exact snippet.
- No app evidence citation without a typed locator when the source supports one.
- No frontend-built citation deep links.
- No final message with pending verifier status.
- No raw database ids exposed to users.
- No business logic in Next.js BFF routes.
- No duplicated app-search or web-search citation models in the UI.
- No speculative generalized knowledge-graph layer in this cutover.
- Branch exhaustively on claim support states.
- Branch exhaustively on evidence locator variants.
- Escape generated prompt text inline at the use site.
- Tests must verify user-visible behavior and persisted evidence contracts.

## Key Decisions

1. Citations attach to claims, not messages.

   Message-level sources cannot prove which source supports which proposition.
   Claim-level evidence is the minimum professional contract.

2. The backend owns citation truth.

   The model can propose evidence ids, but persisted evidence rows come from
   backend-validated retrievals, contexts, locators, and verifier results.

3. Exact snippets are required.

   Titles and source labels are navigation aids, not evidence. A supported claim
   needs quoted source text or a verified source span.

4. Retrieval status is part of the citation.

   Users and operators must know whether a cited source was merely retrieved,
   selected, or actually included in the prompt.

5. Insufficient evidence is a successful product state.

   "Not enough evidence in this scope" is not an error and not a reason to
   search elsewhere. It is the correct answer when the scoped corpus does not
   support the requested claim.

6. Existing memory source refs are reused, not replaced.

   Conversation memory already has source refs and evidence roles. Claim evidence
   should reuse that shape where possible and add stricter snippet and locator
   requirements.

7. Reader jump targets are part of the evidence contract.

   A citation that cannot navigate to the source location is incomplete for app
   content when the app has a reader for that source type.

8. Verification is required before final rendering.

   Prompt instructions reduce failures but do not make citations trustworthy.
   Final citation UI depends on verifier output, not instruction compliance.

9. Old citation rows are not migrated into synthetic evidence.

   Historical message retrievals can remain for audit, but old source chips are
   not rendered after the cutover. New evidence rows start with new assistant
   turns.

10. This is not a compiled wiki layer.

    Compiled source pages and durable knowledge artifacts may come later. This
    cutover makes chat answers evidence-grade.

## Acceptance Criteria

- New assistant messages have exactly one evidence summary.
- Completed source-grounded messages have no pending verifier state.
- Message rows render claim/evidence UI, not source chips.
- No final citation is rendered from model-authored bracket markers.
- No final citation is rendered directly from a retrieval title or source label.
- Supported claims show exact snippets.
- Supported app-source claims show page, section, offset, timestamp, or message
  location when applicable.
- Supported app-source claims have working jump targets.
- Supported web claims open the cited URL and show web-source metadata.
- Unsupported claims render `partially_supported`, `contradicted`,
  `not_enough_evidence`, or `out_of_scope`.
- Scoped chats do not fall back to unscoped retrieval after no results.
- Scoped insufficient-evidence answers show the scope and retrieval status.
- Retrieval rows record selected and included-in-prompt status.
- Prompt assembly records the retrieval ids and context refs used for evidence.
- Claim evidence rows snapshot exact snippets and locators.
- Claim evidence rejects unauthorized sources.
- Claim evidence rejects out-of-scope app sources.
- Claim evidence rejects snippets that do not match the persisted source
  snapshot.
- EPUB evidence jumps to the cited fragment or offset.
- PDF evidence jumps to the cited page geometry.
- Transcript evidence jumps to the cited timestamp.
- Conversation evidence jumps to the cited message.
- Web evidence remains visually distinct from app evidence.
- Existing old citation chips no longer render after the cutover.
- Existing old retrieval rows are not treated as claim evidence unless new claim
  evidence rows reference them.
- Backend integration tests cover support states and scope boundaries.
- Browser component tests cover claim markers and evidence panels.
- E2E covers source-grounded answer, insufficient evidence, and jump-to-source.
- TypeScript typecheck passes.
- Targeted frontend tests pass.
- Targeted backend tests pass.
- Targeted migration tests pass.

## Observability And Evaluation

Persist or emit metrics for:

- retrieval count,
- selected count,
- included-in-prompt count,
- claim count,
- supported claim count,
- unsupported claim count,
- not-enough-evidence count,
- verifier failure count,
- citation jump failure count,
- scope rejection count,
- user feedback on wrong citation,
- user feedback on missing citation.

Evaluation gates:

- claim coverage: all source-grounded claims have a final support state,
- citation precision: supported citations match exact source snippets,
- citation recall: important supported claims have evidence,
- faithfulness: answer claims are supported by retrieved context,
- answer relevance: answer still addresses the user question,
- refusal quality: insufficient evidence states are used when support is missing.

- `docs/rules/database.md`: new tables use UUID primary keys, `created_at`, no
  exposed DB ids, no speculative indexes, and no database-level cascade cleanup.
- `docs/rules/correctness.md`: source refs, locators, and model sidecar output
  are parsed and validated at ingress.
- `docs/rules/generated-text.md`: prompt XML and generated text boundaries
  escape interpolated values inline.
- `docs/rules/json-values.md`: locators and source refs stay typed JSON values,
  not stringified JSON blobs.
- `docs/rules/module-apis.md`: one primary evidence service and one primary
  claim-alignment path.
- `docs/rules/simplicity.md`: no compatibility mode, no fallback UI, no duplicate
  citation system.
- `docs/rules/testing_standards.md`: behavior tests own the acceptance criteria;
  external LLM calls are mocked only at the provider boundary.
