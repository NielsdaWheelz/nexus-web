# PR-05: Audio Player Controls + Playback Position Persistence

## Goal

Transform the minimal play/pause footer into a full podcast player with scrubber, skip controls, speed control, volume, and durable playback position persistence so users can resume episodes across sessions.

## Builds On

PR-04 (Active Subscription Polling + Ops Hardening) merged. GlobalPlayerFooter and globalPlayer context exist with basic play/pause/seek/track-switching. No playback position is persisted — all state is ephemeral in-memory React state.

## Problem

The current player is unusable for real podcast listening:

1. **No scrubber.** Users cannot see where they are in an episode or drag to seek. The only visual is a raw timecode.
2. **No skip forward/back.** The single most-used podcast control (skip 15s/30s) does not exist.
3. **No speed control.** Podcast listeners routinely use 1.5x–2x. The native `<audio>` element supports `playbackRate` but it's not exposed.
4. **No volume control.** Desktop users expect a volume slider.
5. **No position persistence.** Navigating away or refreshing loses your place. For a 2-hour episode this is a dealbreaker. `ReaderMediaState` exists for EPUB/PDF reading position but nothing equivalent exists for audio.

## Scope

### Backend

- **New model: `PodcastListeningState`** — per-user, per-media playback position tracking.
  - Columns: `user_id`, `media_id`, `position_ms` (INT, NOT NULL, default 0), `duration_ms` (INT, nullable), `playback_speed` (FLOAT, NOT NULL, default 1.0), `updated_at` (TIMESTAMPTZ).
  - Primary key: `(user_id, media_id)`.
  - Constraint: `position_ms >= 0`.
  - This is a write-heavy, read-seldom table. Position updates are frequent (every ~10-30 seconds during playback).
- **New endpoints:**
  - `PUT /media/{id}/listening-state` — upsert position + speed. Body: `{ position_ms, duration_ms?, playback_speed? }`. Idempotent. Returns 204.
  - `GET /media/{id}/listening-state` — returns current position + speed for the viewer. Returns 200 with state or 200 with defaults (position 0, speed 1.0) if no row exists.
- **Extend `MediaOut` schema** — add optional `listening_state: { position_ms, playback_speed }` to media detail responses so the frontend can hydrate resume position on page load without a second request.
- **Migration** — new table `podcast_listening_states`.

### Frontend

- **Seek bar / scrubber** — horizontal progress bar in `GlobalPlayerFooter` showing elapsed/total with drag-to-seek. Clicking anywhere on the bar seeks to that position. A filled region shows progress. Buffered region shown as lighter fill.
- **Skip forward / skip back buttons** — configurable skip intervals (default: -15s back, +30s forward). Buttons flanking the play/pause button. Keyboard shortcuts: left arrow (-15s), right arrow (+30s) when player is focused.
- **Speed control** — button/dropdown showing current speed (e.g., "1.5x"). Options: 0.5, 0.75, 1.0, 1.25, 1.5, 1.75, 2.0, 2.5, 3.0. Sets `audio.playbackRate`. Persisted to `PodcastListeningState.playback_speed` so it restores on resume.
- **Volume control** — slider (desktop only; mobile uses hardware volume). Sets `audio.volume`. Persisted to localStorage (not per-episode — this is a global preference).
- **Position persistence** — debounced writes to `PUT /media/{id}/listening-state` every 15 seconds during active playback and on pause/track-change/page-unload (`beforeunload`). On `setTrack()`, hydrate `seek_seconds` from the listening state included in `MediaOut`.
- **Resume UX** — when opening a media page for an episode with a saved position > 0, show a "Resume from MM:SS" prompt or auto-seek (prefer auto-seek with a brief toast: "Resuming from 45:30").

### UI Layout

The footer expands from its current minimal state:

```
┌──────────────────────────────────────────────────────┐
│   advancement: episode title                          │
│  ◄◄ 15s   ▶ / ❚❚   30s ►►   ═══════●══════  23:45 / 1:02:30   1.5x   🔊━━━  │
└──────────────────────────────────────────────────────┘
```

## Acceptance

- seek bar renders, reflects current position, and is draggable to seek to arbitrary positions.
- skip back (-15s) and skip forward (+30s) buttons work and clamp to [0, duration].
- speed control changes playback rate immediately and persists across sessions via `PodcastListeningState`.
- volume slider changes audio volume; preference persists in localStorage across sessions.
- playback position is written to `PUT /media/{id}/listening-state` at most once every 15 seconds during active playback, and always on pause, track change, and page unload.
- opening a media page for an episode with a saved position auto-seeks to the saved position and shows a resume toast.
- `GET /media/{id}` (media detail) includes `listening_state` in the response when the viewer has a saved position.
- keyboard shortcuts (left/right arrow) work when the player footer is focused.
- player controls degrade gracefully for non-podcast media (video embeds, etc.) — speed/skip still work for any audio-capable media.
- migration adds `podcast_listening_states` table with correct constraints and indexes.
- position writes are debounced and do not cause excessive backend load (one write per 15s max).
- all new backend endpoints require authentication and scope to the viewer's own state.

## Non-Goals

- no sleep timer in this PR (PR-06 candidate or future).
- no queue/playlist/next-previous (PR-06).
- no played/unplayed episode state (PR-07).
- no mini-player or expanded player modal — footer-only for now.
- no streaming position via SSE/websocket — polling/write-on-change only.

## Key Decisions

**Position write frequency**: 15-second debounce balances data durability against write volume. On a 1-hour episode, that's ~240 writes max. Acceptable for the expected user scale. If this becomes a bottleneck, batch writes or move to localStorage-first with periodic sync.

**Per-episode speed vs global speed**: speed is stored per-episode in `PodcastListeningState` because different podcasts suit different speeds. New episodes default to 1.0x (not the last-used speed). This is a deliberate choice — if users complain, a "default speed" user preference can be added later.

**Volume is global, not per-episode**: volume is a hardware-level concern, not content-level. Stored in localStorage, not the database.

**Footer-only player**: no expanded/modal player view. The footer is the player. This keeps scope tight. If users want a richer player surface, that's a future PR.

## Risks

- `beforeunload` is not reliable on mobile Safari — position may be lost on mobile tab close. Mitigated by the 15-second periodic writes during active playback.
- `playbackRate` behavior varies across browsers for rates above 2x. Some browsers may distort audio. Test on Chrome, Firefox, Safari.
- Write volume to `podcast_listening_states` under concurrent users — monitor and add rate limiting if needed.
