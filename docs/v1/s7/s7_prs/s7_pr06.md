# PR-06: Playback Queue + Auto-Advance + Next/Previous

## Goal

Add a persistent playback queue so users can line up episodes, auto-advance to the next episode when the current one finishes, and navigate forward/backward through the queue.

## Builds On

PR-05 (Audio Player Controls + Playback Position Persistence) merged. The player has full controls (scrubber, skip, speed, volume) and persists playback position. But it only plays one episode at a time with no concept of "what's next."

## Problem

Without a queue, the user must manually navigate to the next episode and click play every time. This breaks the core podcast listening flow:

1. **No auto-advance.** When an episode finishes, silence. The user must manually find and play the next episode. Podcast Addict auto-advances through a queue.
2. **No next/previous buttons.** Even if the user knows what they want to listen to next, there's no way to skip to it from the player.
3. **No "play next" / "add to queue" actions.** On the episode list, the only action is "play this episode" which replaces whatever is currently playing.
4. **No queue visibility.** Users can't see what's coming up or reorder upcoming episodes.

## Scope

### Backend

- **New model: `PlaybackQueueItem`** — ordered queue per user.
  - Columns: `id` (UUID PK), `user_id` (FK), `media_id` (FK), `position` (INT, NOT NULL — sort key), `added_at` (TIMESTAMPTZ), `source` (ENUM: manual, auto_subscription, auto_playlist).
  - Unique constraint: `(user_id, media_id)` — an episode appears at most once in a user's queue.
  - Index on `(user_id, position)` for ordered retrieval.
- **New endpoints:**
  - `GET /playback/queue` — returns the user's queue, ordered by `position`. Includes media title, podcast name, duration, listening state (position_ms if partially listened).
  - `POST /playback/queue/items` — add one or more episodes to the queue. Body: `{ media_ids: UUID[], insert_position?: "next" | "last" }`. "next" inserts after the currently-playing item; "last" appends. Returns updated queue.
  - `DELETE /playback/queue/items/{item_id}` — remove an item from the queue.
  - `PUT /playback/queue/order` — reorder queue. Body: `{ item_ids: UUID[] }` — the full ordered list of item IDs. Server validates all IDs belong to the user and replaces position values.
  - `POST /playback/queue/clear` — clear the entire queue.
  - `GET /playback/queue/next` — returns the next item after the currently-playing media_id (passed as query param `?current_media_id=...`). Used by auto-advance.
- **Extend subscription sync** — when new episodes are ingested via subscription sync, optionally auto-append them to the subscriber's queue (gated by a per-subscription `auto_queue` boolean, default false). This is additive — if the user has auto_queue enabled, new episodes get `source=auto_subscription` queue entries.

### Frontend

- **Next / Previous buttons** — added to `GlobalPlayerFooter` flanking the skip buttons.
  - Next: advances to the next queue item. If no next item, button is disabled/dimmed.
  - Previous: if current position > 3 seconds, restarts current episode. If position ≤ 3 seconds, goes to previous queue item. If no previous item, restarts current.
- **Auto-advance** — when the `<audio>` element fires `ended`, automatically call `GET /playback/queue/next` and `setTrack()` for the next item. If no next item, stop playback (don't loop).
- **Queue panel** — a slide-up or slide-out panel accessible from a queue icon button in the footer.
  - Shows ordered list of upcoming episodes with podcast artwork thumbnail, title, duration.
  - Currently-playing item highlighted at top.
  - Drag-to-reorder (calls `PUT /playback/queue/order` on drop).
  - Swipe-to-remove or X button per item (calls `DELETE /playback/queue/items/{id}`).
  - "Clear queue" action at bottom.
- **"Play Next" / "Add to Queue" actions** — on episode list rows (podcast detail page) and media detail page:
  - "Play Next" → `POST /playback/queue/items` with `insert_position: "next"`.
  - "Add to Queue" → `POST /playback/queue/items` with `insert_position: "last"`.
  - If the episode is already in the queue, show "In Queue" badge instead.
- **Queue count badge** — small count indicator on the queue icon in the footer showing number of upcoming items.

## Acceptance

- `POST /playback/queue/items` with `insert_position: "next"` inserts the episode immediately after the currently-playing item in position order.
- `POST /playback/queue/items` with `insert_position: "last"` appends to the end of the queue.
- adding an episode that's already in the queue returns 200 with the existing queue (no duplicate, no error).
- `PUT /playback/queue/order` reorders all queue items atomically; rejects requests with missing or extra item IDs.
- `DELETE /playback/queue/items/{id}` removes the item and shifts positions; 404 if item doesn't belong to viewer.
- `POST /playback/queue/clear` removes all queue items for the viewer.
- next button advances to the next queue item and begins playback; disabled when no next item exists.
- previous button restarts if position > 3s, otherwise goes to previous queue item; disabled (restart only) when no previous item exists.
- auto-advance fires when `<audio>` `ended` event triggers; loads and plays the next queue item automatically. If queue is empty, playback stops.
- queue panel opens from footer, shows all items in order with drag-to-reorder and remove actions.
- queue panel reflects real-time state (items added/removed from other tabs or auto-subscription sync update the list on next open).
- reorder via drag-and-drop calls `PUT /playback/queue/order` and updates UI optimistically.
- "Play Next" and "Add to Queue" actions appear on episode list rows and media detail page.
- episodes already in the queue show an "In Queue" badge instead of add actions.
- migration adds `playback_queue_items` table with correct constraints and indexes.
- queue is scoped to the authenticated user; no cross-user leakage.
- queue survives page refresh (server-side persistence, not localStorage).

## Non-Goals

- no smart queue generation (auto-populating queue from subscriptions based on recency/priority) — only manual add and optional auto_queue per subscription.
- no shuffle mode.
- no repeat mode (single or queue).
- no queue sync across devices in real-time (eventual consistency via server persistence is sufficient).
- no "Up Next" vs "Queue" distinction (Podcast Addict separates these; we use a single ordered list).

## Key Decisions

**Single queue, not playlists**: the playback queue is ephemeral and operational ("what am I listening to now/next"). Libraries serve the playlist/collection role. The queue is not a library and not persisted as one.

**Server-side queue persistence**: the queue lives in the database, not localStorage, so it survives across devices and browser sessions. Write frequency is low (user actions only, not periodic), so DB load is minimal.

**No duplicate episodes in queue**: `(user_id, media_id)` uniqueness prevents the same episode from appearing twice. "Play Next" on an already-queued episode moves it to the next position rather than adding a duplicate.

**Previous button behavior**: the 3-second threshold for restart-vs-go-back matches Podcast Addict, Spotify, and most music/podcast players. It's intuitive — if you just started an episode, previous goes back; if you're mid-episode, it restarts.

## Risks

- drag-to-reorder on mobile is finicky — test touch interactions carefully. Consider using a proven library (e.g., @dnd-kit).
- queue reorder sends the full ordered list of IDs, which is simple but doesn't scale to thousands of items. For podcast queues this is fine (unlikely to exceed ~100 items).
- auto-advance across page navigations: if the user navigates to a non-media page while listening, auto-advance still needs to work since the player is in the layout. Verify the `ended` listener persists across route changes.
