# PR-12: Episode Show Notes + Batch Transcript Request

## Goal

Surface episode descriptions (show notes) in the UI and allow users to request transcripts for multiple episodes in a single action.

## Builds On

PR-07 (Episode State Tracking) merged. Episode list has state badges, filtering, and sorting. Transcript request works per-episode with budget forecasting.

## Problem

1. **No episode show notes.** Every podcast app shows episode descriptions — often rich HTML with links, timestamps, guest bios, and sponsor info. The data exists in the RSS feed and is fetched during sync, but it's not stored or surfaced. Users clicking into an episode see the transcript (if available) but no editorial context.
2. **No batch transcript request.** Users subscribing to a new podcast may want to transcribe the 5 most recent episodes at once. Currently they must click "Request transcript" on each episode individually, selecting a reason each time. Tedious for onboarding flows.

## Scope

### Backend — Episode Show Notes

- **Extend `PodcastEpisode` model** — add `description_html` (TEXT, nullable) and `description_text` (TEXT, nullable) columns.
  - `description_html`: raw HTML from RSS `<content:encoded>` or `<description>` (sanitized — strip scripts, event handlers, dangerous tags; allow links, formatting, images).
  - `description_text`: plain text fallback (strip all HTML, collapse whitespace). Used for search indexing and plain-text contexts.
- **Populate during sync** — in `run_podcast_subscription_sync_now`, extract description from feed item and persist. If `<content:encoded>` exists, prefer it over `<description>` (richer content). Sanitize HTML using the same sanitization pipeline as web article fragments (reuse `sanitize_html` if it exists, or a shared utility).
- **Extend episode list response** — add `description_html` and `description_text` to episode list items. For list views, truncate `description_text` to 300 chars to keep response sizes reasonable. Full content available on media detail.
- **Extend media detail response** — `GET /media/{id}` includes `description_html` (full, sanitized) when the media is a podcast episode.
- **Migration** — add columns to `podcast_episodes` table.
- **Input validation** — limit `description_html` to 100KB and `description_text` to 50KB. Truncate silently if RSS provides more (some feeds embed entire blog posts).

### Backend — Batch Transcript Request

- **New endpoint: `POST /media/transcript/request/batch`** — request transcripts for multiple episodes in one call.
  - Body: `{ media_ids: UUID[], reason: string }`.
  - For each media_id: run the same admission logic as single-episode request (quota check, reservation, enqueue).
  - Process sequentially within the request to maintain quota accounting accuracy (each admission decrements the remaining pool for the next).
  - Returns per-episode results: `{ results: [{ media_id, status: "queued" | "already_ready" | "already_queued" | "rejected_quota" | "rejected_invalid", required_minutes?, remaining_minutes?, error? }] }`.
  - Stop processing if quota is exhausted (remaining episodes get `rejected_quota` without individual quota checks).
  - Max 20 episodes per batch (prevent abuse).
- **Reuse existing admission logic** — call `request_podcast_transcript_for_viewer` in a loop, not duplicated logic. Each call runs its own quota check against the decremented pool.

### Frontend — Show Notes

- **Episode list** — below each episode title, show a 2-line truncated plain text preview of `description_text` in muted text. Clicking "Show more" expands to full text (still in the list row, not a modal).
- **Media detail page** — in a "Show Notes" section/tab below the chapter list (or in a collapsible panel):
  - Render `description_html` (sanitized server-side) in a styled container.
  - Links open in new tabs (`target="_blank" rel="noopener"`).
  - Images render inline (with max-width: 100%).
  - If no `description_html`, fall back to `description_text` in a `<pre>` or formatted paragraph.
  - If neither exists, show nothing (no empty state).
- **Timestamp links in show notes** — many podcast show notes contain timestamps like "12:30 - Interview starts". Detect patterns matching `HH:MM:SS` or `MM:SS` and render them as clickable links that seek the player to that timestamp. Regex: `/\b(\d{1,2}):(\d{2})(?::(\d{2}))?\b/g`. Only activate this for podcast episodes with an active player track.

### Frontend — Batch Transcript Request

- **"Transcribe unplayed" button** — on the podcast detail page header (next to "Mark all as played"):
  - Label: "Transcribe all" or "Transcribe unplayed".
  - On click: opens a confirmation dialog showing:
    - Number of eligible episodes (unplayed/in-progress without transcripts).
    - Total estimated minutes (sum of durations).
    - Remaining daily quota.
    - Whether the batch fits within budget.
  - On confirm: POST to `/media/transcript/request/batch` with the eligible media_ids.
  - Results displayed inline: "5 queued, 2 already ready, 1 rejected (quota)".
  - Refresh episode list after completion to reflect new transcript states.
- **BFF route** — `POST /api/media/transcript/request/batch` proxying to backend.

## Acceptance

- episode list rows show a 2-line truncated description preview below the title.
- "Show more" expands the preview to full text within the row.
- media detail page shows sanitized HTML show notes with working links and inline images.
- show notes HTML is sanitized server-side: no scripts, event handlers, or dangerous tags.
- timestamps in show notes (e.g., "12:30") are clickable and seek the player when the episode is playing.
- `description_html` and `description_text` are populated during subscription sync from RSS feed data.
- `<content:encoded>` is preferred over `<description>` when both exist.
- description fields are truncated to 100KB/50KB limits.
- batch transcript request accepts up to 20 media_ids and returns per-episode results.
- batch request processes sequentially and stops early when quota is exhausted.
- remaining episodes after quota exhaustion get `rejected_quota` status without wasting computation.
- "Transcribe all" button shows a confirmation dialog with cost estimate before proceeding.
- batch results are displayed inline showing queued/ready/rejected counts.
- episode list refreshes after batch request to reflect updated transcript states.
- migration adds `description_html` and `description_text` columns to `podcast_episodes`.
- existing episodes without descriptions show no empty state (graceful absence).

## Non-Goals

- no show notes editing by users.
- no show notes search (could be added to keyword search later).
- no show notes in search result snippets.
- no batch transcript request from the subscriptions page (only from podcast detail).
- no automatic background transcription (batch is user-initiated only).

## Key Decisions

**Server-side HTML sanitization**: show notes HTML is sanitized on write (during sync), not on read. This means the `description_html` column contains safe HTML that the frontend can render via `dangerouslySetInnerHTML` without additional client-side sanitization. Same approach as web article fragments.

**Sequential batch processing**: processing media_ids sequentially (not in parallel) within the batch request ensures quota accounting is correct — each admission sees the true remaining balance after the previous admission's reservation. Parallel processing would require pessimistic locking or risk over-commitment.

**20-episode batch limit**: prevents a single request from reserving hours of quota. A user with 60 minutes daily quota requesting 20 × 30-minute episodes would exhaust quota on the first 2 and get 18 rejections — the limit prevents wasting server time on obviously-doomed requests.

**Timestamp detection in show notes**: regex-based, activated only for podcast episodes. False positives (e.g., "12:30 PM" time-of-day references) are low-risk — clicking a false positive just seeks the player, which is harmless and easily undone.

## Risks

- RSS `<description>` and `<content:encoded>` content varies wildly across feeds. Some include full blog posts with embedded videos, iframes, and complex CSS. The sanitizer must be aggressive — strip everything except basic formatting tags (p, a, strong, em, ul, ol, li, br, img).
- Show notes HTML rendering can break page layout if the content includes absolute positioning or wide elements. Use a container with `overflow: hidden` and `max-width: 100%`.
- Batch transcript request could hammer Celery with 20 simultaneous transcription jobs. The jobs are enqueued individually and the worker processes them at its own pace (concurrency=1), so this is naturally throttled.
- Backfilling `description_html` for existing episodes requires a re-sync. No migration backfill — descriptions populate on next sync.
