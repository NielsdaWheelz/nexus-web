# PR-04: Active Subscription Polling Orchestration + Ops Hardening

## Goal

Make active-subscription polling continuously operational in production, with reliable run controls and operator-grade observability.

## Builds On

PR-03 (Real Podcast Transcription Pipeline + Transcript Invariants) merged.

## Acceptance

- active-subscription polling runs automatically on a configured background schedule (worker/beat path), not only via direct helper invocation.
- scheduled polling execution is singleton-safe at the cluster level so overlapping scheduler instances do not run duplicate passes.
- each poll pass evaluates only subscriptions in `active` state; unsubscribed subscriptions never receive future auto-ingest from scheduled polling.
- each run is bounded by explicit run limits so one pass cannot grow unbounded with total subscription count.
- duplicate or overlapping poll runs remain idempotent and concurrency-safe: claim semantics prevent double-ingest and duplicate transcription work for the same subscription episode set.
- stale `running` sync claims are recoverable after a configured lease timeout so crashed workers do not permanently strand subscriptions.
- poll execution emits structured operator telemetry for every run: `processed_count`, `failed_count`, `skipped_count`, and `scanned_count`.
- poll outcomes are durably queryable per run (with deterministic counters and stable failure-code breakdown) and correlated with structured logs for live triage.
- failed sync outcomes exposed through polling use stable error-code semantics suitable for triage and alerting.
- unsubscribe lifecycle guarantees remain intact under scheduled polling, including explicit mode behavior and no implicit removals from shared libraries.
- visibility and processing-state regression suites remain green with scheduled-polling scenarios included.

## Non-Goals

- no recommendation/ranking behavior changes.
- no advanced audio player feature expansion.
- no semantic/vector podcast search work.
- no billing product-surface expansion.

## Key Decisions (if any beyond L2)

**Orchestration Boundary**: scheduled polling reuses the established subscription-sync contract so idempotency, quota, and transcript failure semantics stay single-sourced.

**Concurrency Claim Source of Truth**: subscription sync lifecycle state remains the authoritative claim mechanism across concurrent pollers/workers.

**Lease-Based Recovery**: `running` sync state is treated as a time-bound lease, not a permanent lock, and expired claims are safely reclaimable.

**Operator Triage Surface**: polling observability is contract-level, with deterministic run counters and stable failure-code reporting persisted per run and emitted in structured logs.
