# PR-13: Per-Subscription Settings (Default Speed, Auto-Queue, Notifications)

## Goal

Add per-subscription settings so users can configure default playback speed and automatic queue behavior per podcast, surfaced through a subscription settings panel.

## Builds On

PR-06 (Queue) and PR-07 (Episode State) merged. The `auto_queue` boolean exists on `PodcastSubscription` in the backend but has no UI. Playback speed is per-episode in `PodcastListeningState` but always defaults to 1.0x.

## Problem

1. **No per-podcast default speed.** Users who listen to 10 podcasts at different speeds (1.0x for storytelling, 1.75x for news, 2.0x for tech talks) must re-set speed on every single episode. Podcast Addict lets you set a default speed per podcast — new episodes inherit it.
2. **No auto-queue UI.** The `auto_queue` boolean on `PodcastSubscription` exists in the database and the sync pipeline respects it, but there's no way to toggle it. Users can't say "automatically add new episodes from this podcast to my queue."

## Scope

### Backend

- **Extend `PodcastSubscription` model** — add `default_playback_speed` (FLOAT, nullable, default null). When null, falls back to 1.0x. Valid range: 0.5–3.0.
- **New endpoint: `PATCH /podcasts/subscriptions/{podcast_id}/settings`** — update per-subscription settings.
  - Body: `{ default_playback_speed?: number | null, auto_queue?: boolean }`.
  - Validates speed range (0.5–3.0) if provided. `null` clears the override (fall back to default).
  - Returns updated subscription settings.
- **Extend subscription detail response** — include `default_playback_speed` and `auto_queue` in `PodcastSubscriptionStatusOut` and `PodcastSubscriptionListItemOut`.
- **Extend episode list response** — include `subscription_default_playback_speed` on each episode so the frontend knows what speed to use when starting a new episode.
- **Wire default speed into listening state** — when `setTrack()` is called for an episode with no existing `PodcastListeningState` (first play), the frontend should use `subscription_default_playback_speed` as the initial speed (passed via `setTrack({ playback_rate })`) instead of hardcoded 1.0x.
- **Migration** — add `default_playback_speed` column to `podcast_subscriptions`.

### Frontend

- **Subscription settings panel** — accessible from:
  - A gear/settings icon on each subscription row in the subscriptions list.
  - A settings section on the podcast detail page header.
- **Settings panel contents:**
  - **Default playback speed** — dropdown matching the player speed options (0.5x, 0.75x, 1.0x, 1.25x, 1.5x, 1.75x, 2.0x, 2.5x, 3.0x) plus "Default (1.0x)" as the null/clear option.
  - **Auto-queue new episodes** — toggle switch with label: "Automatically add new episodes to my queue". Shows explanation text: "New episodes from this podcast will be added to the end of your playback queue when they're synced."
  - Save button (or auto-save on change with debounce).
- **Speed inheritance on first play** — when user clicks play on an episode with no existing listening state:
  1. Check `subscription_default_playback_speed` from the episode data.
  2. If set, pass to `setTrack({ playback_rate: subscription_default_playback_speed })`.
  3. If null, use 1.0x (existing behavior).
  4. Once the user manually changes speed on an episode, the per-episode speed in `PodcastListeningState` takes precedence on future resumes.
- **Visual indicator** — on the podcast detail page, show the current default speed and auto-queue state as a summary line: "1.5x default speed · Auto-queue on".

### BFF Route

- `PATCH /api/podcasts/subscriptions/{podcastId}/settings` proxying to backend.

## Acceptance

- user can set a default playback speed (e.g., 1.5x) for a specific podcast subscription.
- new episodes from that podcast start playing at 1.5x when first opened (no prior listening state).
- episodes with an existing listening state resume at their per-episode saved speed, ignoring the subscription default.
- setting default speed to "Default (1.0x)" / null clears the override.
- user can toggle auto-queue on/off per subscription.
- when auto-queue is on, newly synced episodes are appended to the playback queue with `source=auto_subscription`.
- when auto-queue is off, new episodes are not auto-queued (existing behavior).
- subscription settings are accessible from both the subscriptions list and podcast detail page.
- settings panel shows current values and saves on change.
- `PATCH /podcasts/subscriptions/{podcast_id}/settings` validates speed range (0.5–3.0) and rejects out-of-range values.
- subscription list and detail responses include `default_playback_speed` and `auto_queue` fields.
- migration adds `default_playback_speed` column to `podcast_subscriptions`.

## Non-Goals

- no global default speed (across all podcasts) — that's a user preference, not a subscription setting.
- no per-subscription notification preferences (email, push) — notifications are a separate feature.
- no per-subscription download settings — downloads are not implemented.
- no auto-queue priority/ordering (e.g., "add at front vs end") — always appends to end.

## Key Decisions

**Nullable default speed**: null means "use global default (1.0x)." This is cleaner than storing 1.0 and trying to distinguish "user explicitly chose 1.0" from "user never set a preference." If we later add a global user-level default speed, null means "inherit from global" which is the right cascade: episode > subscription > global > 1.0x.

**PATCH, not PUT**: subscription settings is a partial update — the user may change speed without touching auto_queue, or vice versa. PATCH semantics (only update provided fields) prevent accidental resets.

**Frontend inherits speed, not backend**: the backend stores the subscription default but doesn't inject it into listening state creation. The frontend reads `subscription_default_playback_speed` from the episode data and passes it to `setTrack()`. This keeps the backend simpler (no implicit state creation) and gives the frontend full control over the playback initialization flow.

## Risks

- if a user sets 2.5x on a podcast and forgets, new episodes will play at 2.5x which may be disorienting. The speed indicator in the player footer mitigates this — the speed is always visible.
- auto-queue with a prolific podcast (daily episodes) could fill the queue with 30 episodes/month. The queue panel's "Clear queue" action and per-item remove mitigate this.
