# PR-09: Podcast Chapter Support

## Goal

Parse, store, and display podcast chapter markers so users can navigate long episodes by topic/section, with chapter integration into the player timeline and transcript view.

## Builds On

PR-05 (Player Controls + Position Persistence) merged. The player has a scrubber and seek capability. Transcript segments with timing already exist from PR-03.

## Problem

Many podcasts embed chapter markers in their audio files (MP4/M4A chapters, ID3 chapters) or in their RSS feeds (Podcasting 2.0 `<podcast:chapters>` tag, `<psc:chapters>` namespace). These chapters provide a table of contents for long-form episodes (2+ hours), letting listeners jump to specific topics.

Without chapter support, the only navigation options are: the scrubber (imprecise for long episodes), transcript click-to-seek (requires a transcript), and skip forward/back (tedious for large jumps).

## Background: Chapter Sources

Chapters can come from three sources, in priority order:

1. **RSS feed chapters** — Podcasting 2.0 `<podcast:chapters url="..." type="application/json+chapters" />` tag pointing to a JSON chapters file, or inline `<psc:chapters>` (Podlove Simple Chapters) elements in the feed item. This is the richest source, often with chapter artwork and URLs.
2. **Embedded audio chapters** — MP4/M4A chapter atoms or ID3v2 CHAP frames baked into the audio file. Requires downloading and parsing the audio file headers.
3. **Transcript-derived sections** — for episodes without explicit chapters, long silence gaps or speaker changes in the transcript could be used to generate synthetic chapter boundaries. (Out of scope for this PR — noted as future work.)

## Scope

### Backend

- **New model: `PodcastEpisodeChapter`** — chapters per episode.
  - Columns: `id` (UUID PK), `media_id` (FK to media), `chapter_idx` (INT, NOT NULL — order within episode), `title` (TEXT, NOT NULL), `t_start_ms` (INT, NOT NULL), `t_end_ms` (INT, nullable — inferred from next chapter's start if absent), `url` (TEXT, nullable — link associated with chapter), `image_url` (TEXT, nullable — chapter artwork), `source` (ENUM: rss_podcasting20, rss_podlove, embedded_mp4, embedded_id3), `created_at` (TIMESTAMPTZ).
  - Unique constraint: `(media_id, chapter_idx)`.
  - Index on `(media_id, t_start_ms)`.
- **Chapter extraction during sync** — extend `run_podcast_subscription_sync_now` to:
  1. Check RSS feed item for `<podcast:chapters>` tag. If present, fetch the JSON chapters file and parse.
  2. Check RSS feed item for `<psc:chapters>` (Podlove). If present, parse inline chapter elements.
  3. Store chapters with `source=rss_podcasting20` or `rss_podlove`.
  4. If no RSS chapters found, flag the episode for embedded chapter extraction (deferred — see below).
- **Embedded chapter extraction** (optional, lower priority) — extend `podcast_transcribe_episode_job` or create a separate task to:
  1. After downloading the audio file for transcription, probe for MP4 chapter atoms or ID3 CHAP frames (using ffprobe or mutagen).
  2. If found, store chapters with `source=embedded_mp4` or `embedded_id3`.
  3. Only runs if no RSS chapters already exist for the episode.
- **Extend episode detail response** — `GET /podcasts/{id}/episodes` and media detail include `chapters: [{ chapter_idx, title, t_start_ms, t_end_ms, url, image_url }]` when chapters are available.
- **Extend media detail response** — `GET /media/{id}` includes `chapters` array.
- **Migration** — new table `podcast_episode_chapters`.

### Frontend

- **Chapter list in media page** — on the media detail page for podcast episodes with chapters:
  - A "Chapters" section/tab showing a numbered list of chapters with title and timestamp.
  - Clicking a chapter seeks the player to `t_start_ms` and begins playback.
  - Currently-playing chapter is highlighted based on `currentTimeSeconds`.
  - If the chapter has a `url`, the title is a link that opens in a new tab.
  - If the chapter has `image_url`, show a small thumbnail.
- **Chapter markers on scrubber** — small tick marks on the player scrubber at each chapter boundary. Hovering a tick shows the chapter title as a tooltip.
- **Chapter indicator in player footer** — below the episode title, show the current chapter title (updates as playback progresses through chapters). Format: "Chapter 3: Interview with Guest".
- **Chapter navigation** — optional: long-press or double-tap the next/previous buttons to skip to next/previous chapter instead of next/previous episode. (Alternatively, this can be a separate "chapter forward/back" button pair if the UX gets too overloaded.)
- **Chapter + transcript integration** — in TranscriptMediaPane, show chapter headings inline between transcript segments at the appropriate timestamps. A thin divider with the chapter title marks where each chapter begins in the transcript flow.

## Acceptance

- episodes with Podcasting 2.0 `<podcast:chapters>` in the RSS feed have chapters extracted and stored during sync.
- episodes with Podlove `<psc:chapters>` in the RSS feed have chapters extracted and stored during sync.
- chapters are returned in the episode detail and media detail API responses, ordered by `chapter_idx`.
- chapter list on the media page shows all chapters with title and timestamp.
- clicking a chapter in the list seeks the player to that chapter's start time.
- the currently-playing chapter is highlighted in the chapter list as playback progresses.
- chapter tick marks appear on the scrubber at chapter boundary positions; hovering shows chapter title.
- the player footer shows the current chapter title below the episode title.
- chapter headings appear inline in the transcript view at the correct timestamp positions.
- chapters with URLs render as clickable links.
- chapters with image_url show thumbnails.
- episodes without chapters show no chapter UI (graceful absence, not error states).
- `(media_id, chapter_idx)` uniqueness constraint prevents duplicate chapters.
- re-syncing a podcast does not create duplicate chapters (idempotent upsert by `(media_id, chapter_idx)`).
- migration adds `podcast_episode_chapters` table.

## Non-Goals

- no embedded audio chapter extraction in v1 (RSS-only; embedded is stretch goal).
- no user-created chapters or chapter editing.
- no synthetic chapter generation from transcript analysis.
- no chapter-level bookmarking or notes.
- no chapter artwork as scrubber background (some apps do this; too much UI complexity).

## Key Decisions

**RSS chapters first, embedded later**: RSS chapter parsing is cheap (no audio download needed) and covers a growing percentage of podcasts thanks to Podcasting 2.0 adoption. Embedded chapter extraction requires audio download + parsing, which is heavier and can be added in a follow-up.

**Chapters are episode-level, not transcript-level**: chapters are metadata about the episode's content structure, independent of whether a transcript exists. They're stored on the media/episode, not on transcript versions.

**Chapter end times are optional**: many chapter formats only specify start times. `t_end_ms` is inferred as "next chapter's start" or "episode end" at display time, not stored as a hard value (though it can be stored if the source provides it).

**No chapter-level seek buttons in v1**: overloading next/previous with chapter navigation adds UX complexity. The chapter list click-to-seek and scrubber tick marks are sufficient for v1. Chapter skip buttons can be added if users request them.

## Risks

- Podcasting 2.0 `<podcast:chapters>` points to an external JSON file that must be fetched. This adds an HTTP request per episode during sync. If the chapters URL is down, fail gracefully and skip chapters (not the whole episode sync).
- chapter timing from RSS may not perfectly align with transcript segment timing (different sources, different precision). Display-time alignment should use nearest-segment matching, not exact ms equality.
- some podcasts have 50+ chapters per episode. UI should handle long chapter lists without degrading the page layout (scrollable list with fixed height).
