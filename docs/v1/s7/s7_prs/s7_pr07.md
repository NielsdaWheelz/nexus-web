# PR-07: Episode State Tracking + Filtering + Sorting + New Episode Indicators

## Goal

Track per-user episode completion state (unplayed / in-progress / played), surface new-episode indicators on the subscriptions page, and add filtering and sorting controls to the episode list.

## Builds On

PR-06 (Queue + Auto-Advance) merged. PR-05 introduced `PodcastListeningState` with `position_ms`. This PR derives completion state from position data and adds explicit user overrides.

## Problem

1. **No played/unplayed state.** Users can't tell which episodes they've already listened to. Every episode looks the same in the list.
2. **No "mark as played" action.** Users can't bulk-dismiss old episodes or mark an episode played without actually listening to it.
3. **No new episode count.** The subscriptions page shows podcast names but no indicator of how many new/unlistened episodes are available. Users don't know which podcasts have fresh content.
4. **No filtering.** The episode list shows everything in reverse chronological order. Users can't filter to just unplayed episodes or search within a podcast.
5. **No sorting options.** Some users prefer oldest-first (catch up in order). The only option is newest-first.

## Scope

### Backend

- **Extend `PodcastListeningState`** — add `is_completed` (BOOLEAN, NOT NULL, default false) column.
  - Auto-set to `true` when `position_ms >= duration_ms * 0.95` (within 5% of the end, to account for trailing silence/credits). This is computed on position write, not a DB trigger.
  - Users can manually override: `PUT /media/{id}/listening-state` accepts optional `is_completed` to mark as played/unplayed regardless of position.
- **Derived episode state** — computed from `PodcastListeningState`:
  - `unplayed`: no `PodcastListeningState` row exists, OR `position_ms = 0` AND `is_completed = false`.
  - `in_progress`: `position_ms > 0` AND `is_completed = false`.
  - `played`: `is_completed = true`.
- **Extend episode list endpoint** — `GET /podcasts/{id}/episodes` gains query params:
  - `state`: filter by `unplayed`, `in_progress`, `played`, or `all` (default: `all`).
  - `sort`: `newest` (default), `oldest`, `duration_asc`, `duration_desc`.
  - `q`: search within episode titles (case-insensitive substring match).
- **Extend episode list response** — each episode in the list response includes:
  - `listening_state: { position_ms, duration_ms, playback_speed, is_completed }` (null if no state).
  - `episode_state: "unplayed" | "in_progress" | "played"` (derived).
- **New endpoint: batch mark** — `POST /media/listening-state/batch` — mark multiple episodes as played/unplayed in one request. Body: `{ media_ids: UUID[], is_completed: boolean }`. Used for "mark all as played" on a podcast.
- **New episode count per subscription** — `GET /podcasts/subscriptions` response extended with `unplayed_count` (INT) per subscription. Computed as count of episodes in the subscription's podcast that are visible to the user and have `episode_state = unplayed`. This is a LEFT JOIN count — episodes with no `PodcastListeningState` row count as unplayed.
- **Subscription list sort** — `GET /podcasts/subscriptions` gains `sort` param: `recent_episode` (default — podcast with newest episode first), `unplayed_count` (most unplayed first), `alpha` (alphabetical).

### Frontend

- **Episode state badges** — on each episode row in the podcast detail page:
  - Unplayed: blue dot indicator (left of title).
  - In-progress: small progress bar under the title showing % complete.
  - Played: dimmed/muted text, no indicator.
- **Filter bar** — above the episode list on the podcast detail page:
  - Pill/tab filters: All | Unplayed | In Progress | Played. Clicking sets `?state=` query param.
  - Sort dropdown: Newest | Oldest | Shortest | Longest.
  - Search input: filters by episode title (debounced, 300ms).
- **"Mark as played" / "Mark as unplayed" actions** — per-episode context action (right-click menu or overflow button):
  - If unplayed/in-progress → "Mark as played" (calls `PUT /media/{id}/listening-state` with `is_completed: true`).
  - If played → "Mark as unplayed" (calls `PUT /media/{id}/listening-state` with `is_completed: false, position_ms: 0`).
- **"Mark all as played" action** — on the podcast detail page header, a button/menu action that calls `POST /media/listening-state/batch` for all visible unplayed episodes. Confirm dialog before execution.
- **Unplayed count on subscriptions page** — each subscription row shows a badge with the unplayed count (e.g., "12 new"). Count of 0 shows no badge.
- **Subscription sort** — dropdown on subscriptions page: Recent Episode | Most Unplayed | A–Z.
- **Auto-completion** — when auto-advance fires (from PR-06) and the just-finished episode reaches 95% position, the `is_completed` flag is set automatically on the position write triggered by the `ended` event.

## Acceptance

- episodes with no listening state show as "unplayed" with a blue dot indicator.
- episodes with `position_ms > 0` and `is_completed = false` show as "in-progress" with a progress bar.
- episodes with `is_completed = true` show as "played" with dimmed styling.
- `is_completed` is auto-set to `true` when `position_ms >= duration_ms * 0.95` on any position write.
- users can manually mark an episode as played without listening (sets `is_completed = true`).
- users can mark a played episode as unplayed (resets `is_completed = false` and `position_ms = 0`).
- "Mark all as played" marks all unplayed episodes for a podcast in one request and updates UI optimistically.
- filter bar filters episodes by state; selecting "Unplayed" shows only unplayed episodes.
- sort dropdown orders episodes by newest/oldest/shortest/longest.
- search input filters episodes by title substring (server-side, debounced).
- subscriptions page shows unplayed count per podcast; count updates when episodes are marked played.
- subscription list can be sorted by recent episode, most unplayed, or alphabetically.
- `GET /podcasts/{id}/episodes?state=unplayed&sort=oldest&q=interview` returns correctly filtered, sorted, and searched results.
- batch mark endpoint validates all media_ids belong to the viewer's visible episodes; rejects unknown/invisible IDs.
- migration adds `is_completed` column to `podcast_listening_states` with default false.

## Non-Goals

- no per-episode "favorite" or "star" feature.
- no automatic "mark as played" for skipped episodes (only for fully-listened via 95% threshold).
- no push notification for new episodes.
- no episode description/show notes display (candidate for a future PR).
- no "smart filter" combining multiple criteria (e.g., "unplayed AND shorter than 30 min").

## Key Decisions

**95% completion threshold**: the threshold for auto-marking as played is 95% of duration rather than 100% because many podcasts have trailing silence, credits, or ad segments that listeners skip. This matches Podcast Addict's behavior.

**Derived state, not a separate column**: episode state (unplayed/in-progress/played) is computed from `position_ms` and `is_completed`, not stored as a separate enum. This prevents desync between position and state. The `is_completed` flag is the only persisted override — it handles both auto-completion and manual "mark as played."

**Unplayed count is server-computed**: the count is computed via LEFT JOIN on the episode list query, not cached or materialized. For a podcast with ≤1000 episodes this is fast enough. If it becomes a bottleneck, a materialized count can be added later.

**Search is server-side**: episode title search is done server-side to support large episode lists (some podcasts have 1000+ episodes). Client-side filtering would require loading all episodes upfront.

## Risks

- unplayed count query joins `podcast_episodes` × `podcast_listening_states` for every subscription in the list. For users with many subscriptions (50+), this could be slow. Monitor query performance and consider denormalized counts if needed.
- the 95% auto-completion threshold may not work well for very short episodes (< 2 min) where 5% is just a few seconds. Consider a minimum absolute threshold (e.g., max(95%, duration - 30s)).
- "Mark all as played" on a podcast with 500+ episodes sends 500 media_ids in one request. Consider pagination or a podcast-level endpoint (`POST /podcasts/{id}/mark-all-played`) as an optimization.
