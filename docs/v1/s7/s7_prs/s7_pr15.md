# PR-15: Playback Audio Effects (Silence Trimming, Volume Boost, Mono Mix)

## Goal

Add real-time audio processing effects to the podcast player: silence trimming (skip quiet sections), volume boost (normalize quiet podcasts), and mono mixdown (for single-earbud listening).

## Builds On

PR-05 (Player Controls) merged. The player uses a native `<audio>` element with `playbackRate` and `volume` controls. This PR adds Web Audio API processing between the audio source and the output.

## Problem

Podcast Addict's three most-requested audio features after speed control are:

1. **Silence trimming** — podcasts with long pauses, dead air, or segment transitions waste listener time. At 1.5x speed, a 3-second pause is still ~2 seconds of nothing. Trimming or compressing silence can save 5-15% of episode duration.
2. **Volume boost / normalization** — podcasts are mastered at wildly different levels. Switching between a quiet interview podcast and a loud news podcast requires constant volume adjustment. A simple gain boost + limiter normalizes levels.
3. **Mono mixdown** — listeners using one earbud (common while working, commuting) miss stereo content panned to the other channel. Mixing to mono ensures all audio reaches both ears.

## Scope

### Frontend — Web Audio API Pipeline

- **Audio processing graph** — insert a Web Audio API processing chain between the `<audio>` element and the speakers:
  ```
  <audio> → MediaElementSource → [SilenceTrimmer] → [GainBoost] → [MonoMix] → Destination
  ```
  Each node is optional and bypassed when disabled.

- **AudioContext management** — in `globalPlayer.tsx`:
  - Create a single `AudioContext` on first play (not on mount — browsers require user gesture).
  - Create `MediaElementAudioSourceNode` from the `<audio>` element.
  - Connect through the processing chain to `audioContext.destination`.
  - Resume/suspend AudioContext on play/pause to save resources.
  - Handle AudioContext state changes (`suspended`, `running`, `closed`).

### Silence Trimming

- **Approach**: use an `AnalyserNode` to detect RMS volume. When volume drops below a threshold for a sustained period, increase `playbackRate` to skip through silence faster.
  - Not actual skipping (which would break position tracking) — instead, temporarily set `playbackRate` to 4x–8x during silence, then restore the user's chosen speed when audio resumes.
  - This preserves the audio timeline (no gaps in position tracking) while effectively trimming silence.
- **Parameters:**
  - Silence threshold: -45dB RMS (configurable).
  - Minimum silence duration before trimming: 400ms (don't trim natural speech pauses).
  - Silence speed multiplier: 6x (fast enough to be imperceptible as audio, slow enough to not glitch).
- **Implementation:**
  - `requestAnimationFrame` loop monitoring `AnalyserNode.getFloatTimeDomainData()`.
  - Track consecutive below-threshold frames.
  - When sustained silence detected: `audio.playbackRate = silenceSpeed`.
  - When audio resumes above threshold: `audio.playbackRate = userChosenSpeed`.
  - Update MediaSession position state to reflect the user-chosen speed, not the silence speed (so the lock screen scrubber doesn't jump).
- **Indicator**: show a subtle "trimming" label or icon in the footer when silence trimming is actively speeding through a silent section.

### Volume Boost

- **Approach**: `GainNode` with a configurable gain value (1x–3x), followed by a `DynamicsCompressorNode` to prevent clipping.
  - `GainNode.gain.value` = user-selected boost level.
  - `DynamicsCompressorNode` with soft-knee compression prevents distortion at high boost levels.
- **Levels**: Off (1x), Low (+3dB / ~1.4x), Medium (+6dB / ~2x), High (+9dB / ~2.8x).
- **Compressor settings**: threshold -6dB, knee 12, ratio 4:1, attack 0.003s, release 0.25s. These are conservative — audible compression artifacts are worse than no boost.

### Mono Mixdown

- **Approach**: `ChannelMergerNode` or `ChannelSplitterNode` + `GainNode` to mix left and right channels equally.
  - Split stereo → sum L+R → output as mono to both channels.
  - Alternatively, set `audioContext.destination.channelCount = 1` and `channelCountMode = 'explicit'` if the browser supports it (simpler but less portable).
- **Implementation**: create a `ChannelSplitterNode` (2 channels) → two `GainNode`s (each at 0.5) → `ChannelMergerNode` (1 channel) → destination. This avoids clipping from naive L+R summing.

### UI

- **Audio effects button** in `GlobalPlayerFooter` — a small EQ/waveform icon next to the speed dropdown.
- **Effects panel** (dropdown or popover, not modal):
  - **Silence trimming**: toggle switch + "time saved" counter (cumulative seconds trimmed in current episode).
  - **Volume boost**: Off / Low / Medium / High selector (segmented control or dropdown).
  - **Mono audio**: toggle switch.
- **Active indicator**: when any effect is enabled, the effects button shows a small accent dot (similar to the queue count badge).

### Persistence

- **Effects preferences** stored in `localStorage` (not per-episode, not server-side):
  - `podcast_effects_silence_trim: boolean`
  - `podcast_effects_volume_boost: "off" | "low" | "medium" | "high"`
  - `podcast_effects_mono: boolean`
- Restored on mount. These are listening environment preferences (like volume), not content preferences (like speed).

### State in GlobalPlayerContext

- Add `audioEffects: { silenceTrim: boolean; volumeBoost: string; mono: boolean }` to context.
- Add `setAudioEffects(partial)` action.
- Add `isSilenceTrimming: boolean` (true when actively speeding through silence).
- Add `silenceTimeSavedSeconds: number` (cumulative per episode, reset on track switch).

## Acceptance

- enabling silence trimming causes the player to speed through silent sections (>400ms below -45dB) at 6x and restore user speed when audio resumes.
- silence trimming indicator shows in the footer during active trimming.
- "time saved" counter shows cumulative seconds saved in the current episode.
- silence trimming does not break position tracking — `currentTimeSeconds` reflects real audio timeline position.
- volume boost at "Medium" (+6dB) makes a quiet podcast audibly louder without clipping or distortion.
- volume boost uses a compressor to prevent clipping at high levels.
- mono mixdown combines stereo channels; listeners with one earbud hear full audio.
- all effects can be toggled independently and combined (e.g., silence trim + boost + mono simultaneously).
- effects preferences persist in localStorage across sessions.
- effects button shows an active indicator when any effect is enabled.
- effects panel is accessible from the player footer.
- AudioContext is created on first user gesture (play), not on page load (browser autoplay policy compliance).
- enabling/disabling effects does not interrupt current playback (seamless transition).
- effects work across track switches (chain reconnects to new audio source).

## Non-Goals

- no equalizer (multi-band EQ) — too niche, complex UI.
- no noise reduction — computationally expensive, requires ML models or complex DSP.
- no per-podcast effect presets — effects are global listening environment settings.
- no server-side audio processing — all effects are client-side via Web Audio API.
- no "time saved" statistics across episodes (only per-episode counter).

## Key Decisions

**Speed-up, not skip, for silence trimming**: actually removing silent segments would desync the position timeline (the audio file is 60 minutes but plays in 55). Speeding through silence at 6x preserves the timeline while being perceptually equivalent to trimming. Position tracking, chapter markers, transcript sync, and seek all work correctly.

**localStorage, not database**: audio effects are environmental preferences (like system volume). They depend on the listening context (headphones vs speakers, commute vs desk), not the content. No need to sync across devices — the user's effect preferences on their phone may differ from their laptop.

**Conservative compressor**: aggressive compression makes speech sound robotic and fatiguing. The compressor's job is solely to prevent clipping from the gain boost, not to normalize dynamic range. A 4:1 ratio with soft knee and fast release is transparent on speech.

**Single AudioContext**: creating/destroying AudioContexts per track causes audible clicks and wastes resources. One AudioContext for the lifetime of the player, with nodes reconnected on track switch.

## Risks

- **Web Audio API + `<audio>` element interaction**: creating a `MediaElementAudioSourceNode` from an `<audio>` element takes exclusive ownership. The audio can no longer be heard directly — it must flow through the AudioContext graph. If the graph is misconfigured, audio goes silent with no obvious error. Test thoroughly.
- **Silence detection accuracy**: RMS threshold is content-dependent. A podcast with background music will never trigger silence trimming. A podcast recorded in a noisy room may have ambient noise above the threshold. The -45dB default is conservative (only true silence). Users who want more aggressive trimming would need a configurable threshold (future work).
- **Browser compatibility**: Web Audio API is well-supported but `MediaElementAudioSourceNode` has quirks:
  - CORS: the audio source must have CORS headers or the AudioContext will refuse to process it (security restriction). If podcast CDNs don't set CORS, the entire audio pipeline breaks. Fallback: bypass the AudioContext and play directly through the `<audio>` element when CORS fails, with effects disabled.
  - Safari: `AudioContext` may need to be resumed after tab backgrounding. Handle the `statechange` event.
- **CPU usage**: `requestAnimationFrame` loop for silence detection runs continuously during playback. On low-power devices, this could increase battery drain. Consider using `AnalyserNode.fftSize = 256` (minimum) and checking every 100ms instead of every frame.
- **playbackRate interaction**: silence trimming temporarily overrides `playbackRate`. If the user changes speed during a silent section, the user's new speed must be stored and restored when audio resumes, not the pre-silence speed.
