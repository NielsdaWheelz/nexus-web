# PR-14: Subscription Categories (Podcast Folders)

## Goal

Let users organize podcast subscriptions into named categories (folders) for grouping and navigation, mirroring Podcast Addict's folder system.

## Builds On

PR-07 (Episode State + Filtering) merged. Subscriptions page has sorting and unplayed counts. Libraries exist for media organization but are not connected to subscription-level grouping.

## Problem

Users with 20+ subscriptions have a flat list sorted by recency, unplayed count, or alphabetically. There's no way to group related podcasts (e.g., "Tech", "News", "Comedy", "Work"). Podcast Addict, Apple Podcasts, and Pocket Casts all support folders/groups.

Libraries technically exist as a grouping mechanism, but they operate on media (episodes), not subscriptions. A user can't say "put this podcast in my Tech folder" — they'd have to manually move every episode. Subscription categories are a higher-level organizational primitive.

## Scope

### Backend

- **New model: `PodcastSubscriptionCategory`** — categories per user.
  - Columns: `id` (UUID PK), `user_id` (FK), `name` (TEXT, NOT NULL), `position` (INT, NOT NULL — sort order), `color` (TEXT, nullable — hex color for visual grouping), `created_at` (TIMESTAMPTZ).
  - Unique constraint: `(user_id, name)` — no duplicate category names per user.
  - Index on `(user_id, position)`.
- **Extend `PodcastSubscription`** — add `category_id` (FK to `PodcastSubscriptionCategory`, nullable). Null means "uncategorized."
- **New endpoints:**
  - `GET /podcasts/categories` — list all categories for the viewer, ordered by position. Each includes subscription count and aggregate unplayed count.
  - `POST /podcasts/categories` — create a category. Body: `{ name, color? }`. Auto-assigns next position.
  - `PATCH /podcasts/categories/{id}` — update name, color, or position.
  - `DELETE /podcasts/categories/{id}` — delete category. Subscriptions in it become uncategorized (set `category_id = null`), not deleted.
  - `PUT /podcasts/categories/order` — reorder categories. Body: `{ category_ids: UUID[] }`.
  - `PATCH /podcasts/subscriptions/{podcast_id}/settings` (existing) — extend to accept `category_id` (nullable) for assigning a subscription to a category.
- **Extend subscription list response** — include `category: { id, name, color } | null` per subscription.
- **Extend subscription list endpoint** — add `category_id` query param filter. When set, only return subscriptions in that category. When null/absent, return all.
- **Migration** — new table `podcast_subscription_categories`, add `category_id` FK to `podcast_subscriptions`.

### Frontend

- **Category sidebar/tabs** on subscriptions page:
  - Horizontal tab bar or vertical sidebar showing: "All" + each category name (with unplayed count badges).
  - Clicking a category filters the subscription list to that category.
  - "All" shows everything (existing behavior).
  - "Uncategorized" tab shows subscriptions with no category.
  - Categories are color-coded (small dot or left border in the category's color).
- **Category management:**
  - "New category" button — opens inline form with name input and optional color picker.
  - Edit category (rename, change color) via context menu or edit icon on the tab.
  - Delete category — confirmation dialog: "Subscriptions in this category will become uncategorized."
  - Drag-to-reorder category tabs.
- **Assign subscription to category:**
  - On each subscription row: category badge showing current category (or "Uncategorized").
  - Clicking the badge opens a dropdown to assign/reassign to a different category.
  - Also available in the subscription settings panel (from PR-13).
- **Category in podcast detail page** — show current category as a badge in the podcast header. Clickable to change.

### BFF Routes

- `GET /api/podcasts/categories`
- `POST /api/podcasts/categories`
- `PATCH /api/podcasts/categories/{id}`
- `DELETE /api/podcasts/categories/{id}`
- `PUT /api/podcasts/categories/order`

## Acceptance

- user can create a category with a name and optional color.
- user can assign a subscription to a category via dropdown on the subscription row.
- subscriptions page shows category tabs; clicking a tab filters to that category's subscriptions.
- each category tab shows an aggregate unplayed count (sum of unplayed episodes across subscriptions in that category).
- "Uncategorized" tab shows subscriptions with no category.
- "All" tab shows all subscriptions regardless of category.
- categories can be renamed, recolored, and reordered.
- deleting a category sets its subscriptions' `category_id` to null (uncategorized), does not delete subscriptions.
- `(user_id, name)` uniqueness prevents duplicate category names.
- category assignment persists across sessions (server-side).
- subscription list API supports `category_id` filter param.
- migration adds `podcast_subscription_categories` table and `category_id` FK on `podcast_subscriptions`.
- users with 0 categories see no category UI (graceful absence — tabs only appear when ≥1 category exists).

## Non-Goals

- no nested categories (folders within folders).
- no shared/collaborative categories.
- no OPML import/export of categories (OPML has optional `<outline>` groups, but mapping is lossy).
- no auto-categorization (e.g., by podcast genre).
- no category-level settings (e.g., "all podcasts in Tech play at 1.5x") — settings remain per-subscription.

## Key Decisions

**Categories are subscription-level, not library-level**: libraries group episodes (media); categories group subscriptions (podcasts). These are different concerns. A user might have a "Tech" category containing 5 podcasts, but those podcasts' episodes live in the default library alongside everything else. Conflating these would create confusion.

**Soft delete on category deletion**: subscriptions become uncategorized rather than deleted. Destructive cascades (losing subscriptions when deleting a folder) would be a serious UX mistake.

**Color is optional**: not all users want color-coding. The UI works fine without it (plain text tabs). Color is a progressive enhancement for power users.

**Category tabs, not a sidebar**: a horizontal tab bar above the subscription list keeps the layout consistent with the existing page and works well on mobile. A vertical sidebar would eat horizontal space on narrow viewports.

## Risks

- users with many categories (10+) will overflow the horizontal tab bar. Add horizontal scroll with fade indicators, or switch to a dropdown at a threshold (e.g., >8 categories).
- drag-to-reorder on horizontal tabs is awkward on mobile. Consider a dedicated "Manage categories" modal for reordering on small screens.
- aggregate unplayed count per category requires a join across subscriptions → episodes → listening states. For users with many subscriptions, this could be slow. Consider caching or computing alongside the subscription list query.
