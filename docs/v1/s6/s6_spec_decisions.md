# Slice 6 Spec Decisions

## Decision Ledger

| id | question | decision | rationale | fallback/default | owner | due | status |
|---|---|---|---|---|---|---|---|
| S6-D00 | How should Slice 6 be decomposed for L2 authoring? | Use five contract clusters: (1) scope + L0 constraints, (2) PDF lifecycle + `media.plain_text` readiness, (3) PDF highlight/annotation persistence + APIs, (4) quote-to-chat + visibility compatibility, (5) acceptance/traceability hardening. | Mirrors L1 acceptance surfaces and isolates the main unknown (PDF highlight model) so drafting can proceed without hidden assumptions. | n/a | Spec owner | Immediate | Accepted |
| S6-D01 | What is the canonical persistence model for PDF highlights and annotations while preserving S4 visibility and S3 quote-to-chat compatibility? | Use a unified logical `highlights` aggregate with typed anchor submodels: fragment-offset anchors and PDF geometry anchors. Keep `annotations` attached to the logical `highlights.id`; add immutable `anchor_kind` and `anchor_media_id` to support cross-kind visibility and context handling. | Preserves one product concept (`highlight`/`annotation`), avoids second-class PDF context targets, keeps S4 visibility and S3 quote-to-chat extensible, and provides a durable foundation for future anchor types without route/service duplication drift. | Transitional rollout may preserve legacy fragment fields/endpoints during migration, but the canonical S6 contract is anchor-typed logical highlights. | Spec owner + platform + frontend | S6 | Accepted (user-approved 2026-02-22) |
| S6-D02 | What is the canonical geometry coordinate space and normalization/fingerprinting rule for PDF highlight anchors? | Use `geometry_version=1` with canonical page-space points (unrotated `CropBox`, top-left origin, x-right/y-down), 1-based `page_number`, server-side axis-aligned rectangle-quad normalization, 0.001-pt quantization, deterministic quad sorting, SHA-256 `geometry_fingerprint`, and derived `sort_top/sort_left` ordering keys. | Provides renderer-independent storage, stable duplicate identity, deterministic API ordering, and versioned evolution room while keeping S6 implementation practical with PDF.js text-layer rectangle selections. | Future geometry upgrades (e.g., true non-rect quad support) must use a new `geometry_version` and migration/back-compat strategy; v1 rows remain valid. | Spec owner + frontend + platform | S6 | Accepted (user-approved 2026-02-22) |
| S6-D03 | Which existing file-read endpoint should S6 use for PDF.js document fetches? | Reuse `GET /media/{id}/file` signed download path; S6 only tightens/documentalizes PDF.js usage and visibility semantics. | Endpoint already exists, is visibility-gated, and aligns with private storage + signed URL architecture. A second endpoint would increase drift and audit surface. | If PDF.js integration later needs range/proxy behavior, keep route contract stable and evolve server implementation behind the same path if possible. | Spec owner + platform | S6 | Accepted |
| S6-D04 | Should S6 include PDF ingest-from-URL? | No. S6 remains upload-first; PDF ingest-from-URL stays deferred to v2. | L1 roadmap and L0 constitution explicitly exclude it; avoids scope expansion and keeps S6 focused on reading/highlighting/quote correctness. | Ingestion architecture should remain source-agnostic where feasible so v2 URL ingest can converge on the same extraction pipeline outputs. | Spec owner + product | v2 planning | Accepted |
| S6-D05 | How should PDF quote-to-chat derive nearby context from `media.plain_text` using stored `exact` when matches are ambiguous or absent? | Persist PDF quote match metadata on the PDF anchor (`plain_text_match_*`) and page-span index rows (`pdf_page_text_spans`); use `plain_text_match_version=1` page-local literal matching with statuses `unique|ambiguous|no_match|empty_exact|pending`. Quote-to-chat always includes stored `exact` and includes nearby context only for `unique`; all other outcomes degrade safely by omitting nearby context. | Prevents silent wrong-context attachment on repeated phrases, makes outcomes deterministic/auditable, supports asynchronous enrichment when `media.plain_text` becomes ready later, and preserves user trust while keeping v1 implementation practical. | Future matching improvements must use a new `plain_text_match_version` and preserve backward compatibility for v1 persisted match metadata. | Spec owner + platform + frontend | S6 | Accepted (user-approved 2026-02-22) |
| S6-D06 | How should S6 prevent stale PDF quote-context offsets after text-artifact rewrites without adding artifact generation/version fields? | Do not add artifact-generation version binding in S6. Instead, require retry/rebuild paths that rewrite `media.plain_text` and/or `pdf_page_text_spans` to invalidate all PDF highlight quote-match metadata (`plain_text_match_*`) and clear PDF `prefix/suffix` for that media, then recompute via background/lazy deterministic enrichment. Embedding/search retries must not rewrite PDF text artifacts. | Preserves correctness (no silent wrong nearby context) with materially less schema and contract complexity than generation/version binding. Keeps geometry and stored `exact` stable, degrades safely while pending, and constrains retry behavior explicitly. | If post-S6 operator workflows or repair frequency make invalidation-only insufficient, add artifact-generation binding in a later slice and gate quote context on generation equality. | Spec owner + platform | S6 | Accepted (user-approved 2026-02-24) |
| S6-D07 | How should S6 integrate PDF highlights into linked-items given the long-term goal of one pane across object types? | Reuse the existing linked-items pane shell/product surface and row interaction model in S6; add a PDF renderer alignment/measurement adapter for active-page PDF highlights. Do not ship a separate PDF-only pane. Defer full cross-object linked-items unification (documents + conversations + other object types) to a later refactor. | Ships PDF in S6 with minimum UX divergence, preserves existing linked-items behavior investment, and avoids blocking on a broader UI architecture refactor while still moving toward an adapter-based long-term design. | If S6 implementation friction is high, permit a thin internal adapter seam extraction first (HTML/EPUB + PDF only) and postpone broader genericization. | Spec owner + frontend | S6 | Accepted (user-approved 2026-02-24) |
