# The Amanuensis — write tools for the chat agent under origin discipline — Hard Cutover

**Status:** Spec · Rev 1 · 2026-07-07
**Type:** Hard cutover — no legacy code, no fallbacks, no compat shims.

## One-line

The house agent stops being read-only: five additive, origin-marked, undoable
write tools (`add_to_library`, `jot_note`, `create_highlight`, `mint_edge`,
`queue_add`) turn "chat about your library" into "dictate to your librarian" —
under a new `resource_edges` origin `assistant` with one sole writer, per-run
write caps, a visible trust-trail row per write, and per-write Undo.

## 1. Problem (grounded diagnosis)

### 1.1 The most-used agent in the product cannot write one edge

`_CHAT_TOOL_SPECS` (chat_runs.py:179-199) registers `app_search`, `web_search`,
`read_resource`, `inspect_resource` — all read-only. Horizons' entire thesis is
agents as co-authors under provenance; scriptorium §II names this the second
load-bearing absence. The user can *say* "file this under Criticism, dog-ear
that passage, connect these two," and the agent can find all three objects
(`app_search`, `read_resource`) — then must answer in prose because it has no
hands. Every filing is a manual context-switch out of the conversation.

### 1.2 The origin vocabulary was designed for exactly this and is unused for it

`ck_resource_edges_origin` (models.py:579) already enumerates seven writers, and
`synapse` proved the pattern end to end (synapse-resonance-engine.md): a new
`origin` + one sole-writer module + a `snapshot.excerpt` rationale, zero new edge
tables. `resource_graph.policy.EDGE_SHAPE_POLICIES` (policy.py:46) is a per-origin
shape table waiting for its next entry. The provenance graph's own N9 note
(schemas.py:22, edges.py cleanup comments) documents the extension point. Nothing
technical blocks agent writes; nobody has staffed the scribe.

### 1.3 No server-side quote resolution, so "highlight that passage" is unanchorable

`create_highlight_for_fragment` (highlights.py:328) requires
`(fragment_id, start_offset, end_offset)` — offsets the browser computes from the
DOM. `chat_quote.py` only renders. The agent, holding a passage's *text*, has no
way to turn `{exact, prefix, suffix}` into offsets server-side. The algorithm
exists (`pdf_quote_match.compute_match`) but is bound to PDF page spans, not
`fragments.canonical_text`.

## 2. Target behavior (user-facing)

- **The reader (in chat).** "File this under Criticism" files the subject media
  into the Criticism library; the answer shows a **Filed to Criticism** trail row
  with an **Undo** button. "Dog-ear the sentence about entropy" creates an
  anchored highlight at that exact passage; the row reads **Highlighted "…the
  entropy of the system…"**. "Connect these two" mints an `assistant`-origin edge;
  the row reads **Connected *A* ↔ *B*** with the model's one-line rationale. "Jot
  that in today's note" appends a note block to the daily page. "Queue this to read
  next" adds it to the consumption queue.
- **The agent.** It has five new tools, described in the system prompt, and uses
  them only when the user's words ask it to file/annotate/connect/queue. Each tool
  executes immediately and returns the ids it created so the model can reference
  them. If a quote is ambiguous, the tool refuses and the agent asks the user to
  be more specific. After eight writes in one turn, further write tools refuse.
- **Undo.** Every write row in the trust trail carries **Undo**. Pressing it
  reverts *that* tool call's writes exactly (removes the entry / deletes the
  highlight / deletes the block / deletes the edge / removes the queue item) and
  the row greys to **Undone**. Undo is idempotent.
- **Nothing silently reorganizes.** Every write is origin-marked `assistant`,
  visible in the turn it happened, and reversible. There are **no** delete or
  destroy tools; the agent cannot remove or overwrite anything the user made.

## 3. Goals / Non-goals

- **G-1.** Give the chat agent five additive write tools, each ≈60 lines of
  domain code on the existing tool-loop shape (scriptorium §II economics).
- **G-2.** One new edge origin `assistant` with exactly one sole writer
  (`agent_tools/writes.py`), snapshot-excerpt rationale required, mirroring
  synapse's shape discipline.
- **G-3.** Server-side quote resolution (`services/text_quote.py`) so
  `create_highlight` anchors from `{exact, prefix, suffix}` against canonical
  fragment text; ambiguity is a refusal, never a guess.
- **G-4.** Every write recorded as created refs on its `message_tool_calls` row;
  a per-tool-call Undo endpoint reverts through the owning services and stamps
  `reverted_at`.
- **G-5.** Per-run write cap (one constant) + one config flag
  (`ASSISTANT_WRITE_TOOLS_ENABLED`, default true). Writes execute immediately;
  visibility + undo satisfy explicit-UI doctrine.
- **N-1.** No delete/destroy/overwrite tools of any kind — additive only.
- **N-2.** No proposal/confirmation mode, no approval queue, no "dry run" flag
  (D-5).
- **N-3.** No new edge *kinds* — `assistant` uses the existing
  `context/supports/contradicts`.
- **N-4.** No provenance column on `note_blocks` or `library_entries`: dictated
  prose is user-owned; provenance rides the tool trail (D-2).
- **N-5.** No `llm_calls` owner_kind widening — write tools are non-LLM tool
  executions inside an existing `chat_run` (contrast dawn-write, §10).
- **N-6.** No new resource *schemes*; `assistant` reuses existing endpoints.
- **N-7.** No standalone HTTP write endpoints for these tools — the only caller
  is the chat tool loop (the injection boundary, R-1).

### 4.2 The `assistant` origin shape

`assistant` edges are **bare** (no ordinal, no order keys) and carry a mandatory
`snapshot` whose `excerpt` is the model's one-line rationale (mirrors synapse's
`snapshot.excerpt`; optional `title` = target label at mint time). Endpoints are
restricted to the library graph the user can see:

- `source_scheme ∈ {media, page, note_block, highlight}`
- `target_scheme ∈ {media, page, note_block, highlight}`

**D-1** justifies excluding `conversation`/`message`/citation schemes: edges
to/from a conversation are owned by the `citation`/`system`/`context` origins;
the assistant's hand stays inside the durable library graph, similar to synapse.
Unlike synapse — target schemes only `('media','note_block')` (policy.py:27) —
`assistant` admits `page` and `highlight` too: a **widened** copy of synapse's
shape, not an exact one. All four schemes already appear in
`ck_resource_edges_source_scheme` / `_target_scheme` (models.py:588-615), so **no
scheme CHECK is widened** — only the origin CHECK and the snapshot CHECKs.

### 4.3 The write path (per tool)

Each tool's chat-loop branch: parse args → call the matching `writes.py` handler
→ handler enforces cap + visibility, mutates via the owning service, persists the
`message_tool_calls` row with the created refs written into the existing
`result_refs` array as `[{scheme, id}, …]` (D-9 — no new `created_refs` field),
returns `(tool_call_id, created_refs, tool_output_json)` → loop emits the trust
event and appends a `ToolResult`. Handlers never open standalone transactions
beyond the service's; where a reused service commits internally
(`create_highlight_for_fragment`), the handler records the tool row after and
re-arm-prunes on retry (R-3, D-8).

### 4.4 Undo

`undo_tool_call(db, viewer_id, tool_call_id)` reads the row's `result_refs`,
reverts each created ref through its owning service, and stamps `reverted_at`.
Revert-by-scheme: `edge→delete_edge`, library `entry→delete_entry`,
`highlight→delete_highlight`, `note_block→notes.delete_note_block` (**new seam,
F-02** — no single-block deleter exists today: a direct `db.delete(block_row)` +
`enqueue_note_reindex` re-arm in `notes.py`, mirroring `delete_page`). The `queue`
ref type is **not handled until S6** — it does not exist until lectern lands
(F-04); S0–S5 undo covers the four ref types shippable then.
Idempotent: `reverted_at IS NOT NULL` → no-op success. Each revert tolerates an
already-absent target (the user may have deleted it manually). The trail row
renders **Undone** once `reverted_at` is set.

```
           'synapse','document_embed','assistant')
```

**5.2 Widen snapshot-origin CHECK** (`ck_resource_edges_snapshot_origin`):

```
snapshot IS NULL OR origin IN ('citation','synapse','assistant')
```

**5.3 Widen snapshot-has-ordinal CHECK** (`ck_resource_edges_snapshot_has_ordinal`):

```
snapshot IS NULL OR ordinal IS NOT NULL OR origin IN ('synapse','assistant')
```

**5.4 New `ck_resource_edges_assistant_snapshot_excerpt`** (mirror synapse):

```
origin != 'assistant'
OR (snapshot IS NOT NULL AND snapshot ? 'excerpt'
    AND jsonb_typeof(snapshot->'excerpt') = 'string'
    AND btrim(snapshot->>'excerpt') <> '')
```

**5.5 New `ck_resource_edges_assistant_shape`** (mirror synapse shape):

```
origin != 'assistant'
OR (source_scheme IN ('media','page','note_block','highlight')
    AND target_scheme IN ('media','page','note_block','highlight')
    AND source_order_key IS NULL AND target_order_key IS NULL
    AND ordinal IS NULL)
```

Every enumerating CHECK that mentions origins is listed above; **five** are
touched (three widened, two new). `ck_resource_edges_kind`,
`_source_scheme`, `_target_scheme`, `_snapshot_object`, `_ordinal_origin` are
unchanged (assistant carries no ordinal, all four schemes pre-exist, snapshot is
an object). The unique index `uq_resource_edges_context_pair` (models.py:768)
already deduplicates `(user, origin, source, target)` bare pairs — an assistant
`mint_edge` duplicate is rejected in-app before it reaches it.

**5.6 `message_tool_calls.reverted_at`** — new nullable column:

```
ALTER TABLE message_tool_calls ADD COLUMN reverted_at timestamptz NULL;
```

**D-3.** `reverted_at` is a real column, not a `result_refs` field: it is the
undo lifecycle state (mirrors the `dismissed_at`/soft-state pattern used
elsewhere), must be queryable (the cap counter filters on it — D-6, AC-9), and
keeps `result_refs` carrying only refs/payloads, never lifecycle state.

No new tables. `consumption_queue_items` is **owned by lectern** (P-11); this
cutover reads/writes it only through lectern's queue service.

## 8. Key decisions

- **D-1. Assistant endpoints are the library graph, not conversations.**
  Source/target ∈ `{media, page, note_block, highlight}`. *Rejected:* allowing
  `conversation`/`message` endpoints — those are owned by `citation`/`system`
  origins; an agent hand reaching into conversation edges duplicates the citation
  spine and muddies cleanup ownership. The four chosen schemes pre-exist in the
  scheme CHECKs, so no scheme widening is needed.
- **D-2. No provenance column on dictated content.** A note the user dictates is
  the user's prose — they asked for the words. *Rejected:* an `origin` column on
  `note_blocks`/`library_entries` (a parallel provenance store the graph doctrine
  forbids). Provenance lives in the `message_tool_calls` trail: what the agent
  filed, when, and Undo. Only `mint_edge` carries a machine origin, because an
  edge *is* an assertion (mirrors synapse).
- **D-3. `reverted_at` is a column** (see §5.6).
- **D-4. Ambiguous quote is a tool error, never a guess.** `text_quote.resolve`
  returns `unique | ambiguous | no_match` (mirroring `pdf_quote_match`).
  `ambiguous`/`no_match` → the tool returns an error result telling the model to
  quote more text or add surrounding context; it never picks the first hit.
  *Rejected:* "closest match" heuristics — a wrong anchor is worse than a
  clarifying question.
- **D-5. Writes execute immediately; no proposal mode.** Single-user prototype +
  trust-the-model doctrine: everything is origin-marked, visible in the turn, and
  reversible. *Rejected:* a "propose then confirm" mode / approval queue — it
  reintroduces the verifier/guard complexity the product doctrine deletes, and
  buys nothing that visibility + Undo + the cap don't. (Destruction is different:
  there are simply no destructive tools — N-1.)
- **D-6. One write cap constant, one flag.** `ASSISTANT_MAX_WRITES_PER_RUN = 8`
  (module constant in `writes.py`) counts committed write tool calls for the
  assistant message **whose `reverted_at IS NULL`** (undo reclaims budget — AC-9);
  on the ninth the tool refuses. `ASSISTANT_WRITE_TOOLS_ENABLED` (config, default
  true) omits the five ToolSpecs from `_CHAT_TOOL_SPECS` when false. *Rejected:*
  per-tool caps / per-library policy config — sprawl for a single-user prototype.
- **D-7. `writes.py` is one module for all five tools + undo**, not one file per
  tool. The concern is "assistant-authored mutation," and one owner makes the
  sole-writer negative gate (§13) a single-file assertion. Tool *definitions*
  (the ToolSpec dicts) live in `writes.py` too, mirroring how `app_search.py`
  co-locates definition + executor.
- **D-8. `persist_write_tool_call` is a sibling of `persist_tool_call_trace`, not
  a fork.** Both are thin writers over `message_tool_calls` in `chat_run_tools.py`
  (the sole lifecycle owner) sharing its `FOR UPDATE` re-arm-on-retry pattern. A
  separate function — not a call into `persist_tool_call_trace` — is warranted
  because (a) the write payload is a list of created refs, not the read `result`
  object (`uri`/`status`/`body_chars`) that function is typed for, and (b)
  `create_highlight_for_fragment` commits internally, so the write row is recorded
  **after** an intervening commit (the two-commit window, R-3) — a shape
  `persist_tool_call_trace` never faces. Any schema change updates both.
- **D-9. Write tools reuse `result_refs`; no `created_refs` field.**
  `TrustToolCallOut.result_refs` (`list[dict[str, Any]]`, conversation.py:500)
  already carries "what this tool call produced": write rows store created refs as
  `[{scheme, id}, …]`, read rows store `{uri, status, body_chars}` payloads, told
  apart by `tool_name`. *Rejected:* a parallel `created_refs: list[ResourceRefOut]`
  field (the finding's blocker) — there is no `ResourceRefOut` schema, and a second
  field for the same concern duplicates the wire with no authoritative source.

## 9. What dies (exhaustive)

**Nothing dies.** This cutover is additive by design, and that is doctrinally
acceptable because the origin vocabulary was **designed for new writers**
(provenance N9: a new typed relationship arrives as a new origin with a sole
writer). No file, symbol, CSS block, table, or route is deleted.

- **Grepped for dead "read-only" copy to kill:** `rg -ri "assistant cannot
  (modify|write|change)|can'?t (modify|write) your (library|notes)|read-only
  assistant"` over `apps/web/src` and `python/nexus` returns **no product copy**
  (only unrelated docstring substrings). There is no "the assistant cannot modify

## 12. Acceptance criteria (testable)

- **AC-1. "File this under X" round-trip.** A chat turn whose subject is media M
  and text asks to file it under library L results in exactly one
  `library_entries` row `(L, M)` (via `ensure_entry`, dense position), a
  `message_tool_calls` write row with `result_refs=[library:L… / entry ref]`, and
  a **Filed to *L*** trail row with Undo.
- **AC-2. Ambiguous-quote refusal.** `create_highlight` with an `exact` occurring
  ≥2 times in M's canonical text and no disambiguating prefix/suffix creates **no**
  highlight, returns a tool error result, and the model surfaces a
  clarification — asserted by zero new `highlights` rows and a tool `status`
  reflecting the refusal.
- **AC-3. Undo restores prior state exactly.** After any write tool call, calling
  the undo endpoint removes precisely the created rows (entry / highlight +
  anchor + attached note + any `assistant` edge / block / queue item), leaves all
  other state byte-identical, sets `reverted_at`, and a second undo returns 200
  without further change.
- **AC-4. `mint_edge` is origin-marked and grounded.** The created edge has
  `origin='assistant'`, a non-empty `snapshot.excerpt` (the rationale), no
  ordinal/order keys, endpoints among `{media,page,note_block,highlight}` both
  visible to the viewer; a duplicate bare pair is rejected.
- **AC-5. Cap enforced.** The ninth write tool in one assistant turn returns a
  cap-reached error result and performs no mutation; the first eight succeeded.
- **AC-9. Undo reclaims cap budget.** After eight successful writes in one turn,
  undoing write #8 (stamping `reverted_at`) drops the non-reverted count to seven,
  and a subsequent ninth write **succeeds** — proving the cap counts
  `reverted_at IS NULL` rows only (D-6). Paired unit test: the cap counter returns
  7 when one of eight rows has `reverted_at` set.
- **AC-6. Flag off = read-only agent.** With `ASSISTANT_WRITE_TOOLS_ENABLED=false`,
  `_CHAT_TOOL_SPECS` contains only the four read-only tools and no write branch is
  reachable.
- **AC-7. No destructive tools.** No registered ToolSpec exposes a
  delete/remove/overwrite verb (§13 gate).
- **AC-8. Visibility gate.** `add_to_library` into a library the viewer cannot
  administer, and any write touching a resource the viewer cannot read, are
  rejected before mutation.

## 16. Risks

- **R-1. Prompt-injection via document content (HIGH).** A read document could
  contain text instructing the model to file/connect/queue things.
  *Mitigation:* write tools run **only** from the user-visible chat turn loop
  (N-7, no standalone routes); every write is capped (D-6), origin-marked,
  surfaced in the trail, and one-tap reversible; visibility gating (AC-8) means
  the agent can only touch resources the user can already see — an injection can
  reorganize nothing the user couldn't, and nothing silently.
- **R-2. Runaway filing (MEDIUM).** A confused model files many items.
  *Mitigation:* the per-run cap (8) is a hard stop; all writes are one-tap Undo;
  the trail makes a spree visible in the turn it happened.
- **R-3. Two-commit window on highlight create (MEDIUM).**
  `create_highlight_for_fragment` commits internally, so the created-ref record on
  the tool row is a second write. *Mitigation:* the handler records the tool row
  after the highlight commit and re-arm-prunes on retry (mirrors
  `persist_tool_call_start`'s FOR UPDATE re-arm); an orphaned highlight without a
  recorded ref is still user-visible and manually deletable — never lost.
- **R-4. Ambiguous/stale quote anchors (LOW).** Canonical text can differ subtly
  from what the model quotes. *Mitigation:* D-4 refuses on non-unique/no-match
  rather than guessing; the model is instructed to quote more surrounding text.
- **R-5. Undo of a manually-mutated target (LOW).** The user deletes an entry the
  agent filed, then presses Undo. *Mitigation:* each revert tolerates an
  already-absent target and still stamps `reverted_at` (idempotent, AC-3).
- **R-6. Lectern slip (LOW).** If lectern is descoped, `queue_add` (S6) is the
  only slice affected; S0-S5 ship the other four hands independently (§11).
