# Agent App Search Tool

This is the target-state implementation spec for letting the assistant search
Nexus app content for the user.

This is a hard cutover plan. There is no legacy mode, compatibility adapter,
dual context format, or fallback implementation once this work lands.

## Problem

The assistant can currently answer from the prompt, attached contexts, and
conversation history. It cannot reliably find saved Nexus content on behalf of
the user unless the user manually attaches the right item first.

The target behavior is that the assistant can search the user's visible Nexus
content, retrieve bounded source context, and answer with citations and app
deep links.

## Goals

- Give the assistant a read-only `app_search` tool for saved Nexus content.
- Search across media, fragments, annotations, messages, transcript chunks,
  podcast episodes, podcasts, videos, and other first-class saved content.
- Preserve existing authorization guarantees: invisible content is never
  returned, cited, summarized, or hinted at.
- Reuse the existing search capability as the canonical retrieval substrate.
- Keep business logic in FastAPI services. Next.js BFF routes remain transport
  proxies only.
- Return compact, typed, citation-ready search results to the model.
- Fetch and render source context through deterministic backend code, not by
  letting the model decide raw database shape.
- Support streaming and non-streaming chat with the same tool loop semantics.
- Persist enough retrieval metadata to debug, audit, and evaluate answers.
- Keep the implementation provider-agnostic across supported LLM providers.

## Non-Goals

- No external discovery or acquisition search. `/browse` remains separate.
- No OpenAI hosted File Search as the canonical app search backend.
- No raw SQL, database, filesystem, browser, or arbitrary HTTP tool exposed to
  the model.
- No write tools, mutation tools, delete tools, follow/unfollow tools, ingest
  tools, or library-management tools.
- No MCP server as the internal implementation path.
- No compatibility with the old prompt-only chat path for app-search-capable
  models.
- No hidden fallback where the assistant invents citations when retrieval
  fails.

## Target Behavior

When the user asks about saved Nexus content, the assistant searches the app
before answering. Examples:

- "Find the episode where they talked about memory."
- "What did I highlight about embeddings?"
- "Which saved video mentions pgvector?"
- "What did I say earlier about this article?"
- "Search my library for the paper about agentic RAG."

The assistant may run bounded follow-up searches when the first result set is
ambiguous or incomplete. The assistant must stop searching when it has enough
evidence, when it reaches the configured budget, or when no useful results are
found.

Answers that use retrieved app content must include source citations. Citations
must resolve to Nexus deep links, including media timestamps when available.

If the search finds no relevant visible content, the assistant says that it did
not find relevant saved content. It may answer from general knowledge only when
the user asked a general question and the response clearly separates that from
saved Nexus content.

The user-facing stream shows tool activity in a compact form:

- search started
- search query summary
- result count
- cited source chips
- final answer tokens

The model never sees invisible content, hidden IDs for invisible objects, raw
database rows, or unbounded source text.

## Final State

### Search Contract

The canonical model-facing tool is:

```text
app_search(query, scope, types, semantic, limit, cursor)
```

Parameters:

- `query`: natural-language query string, 2 or more non-whitespace characters.
- `scope`: one of `all`, `media:<uuid>`, `library:<uuid>`,
  `conversation:<uuid>`.
- `types`: optional list of typed content classes.
- `semantic`: enables semantic transcript search when transcript-like types are
  requested.
- `limit`: 1 to 20 for model-facing calls.
- `cursor`: opaque pagination cursor returned by prior calls.

Result item shape:

```text
{
  type,
  id,
  title,
  snippet,
  score,
  media_id,
  media_kind,
  source_label,
  published_date,
  authors,
  t_start_ms,
  t_end_ms,
  deep_link,
  context_ref
}
```

Rules:

- `context_ref` is stable and typed. It is not a raw database row.
- `deep_link` is generated by backend-owned or shared canonical link helpers.
- Result text is snippet-sized. Full context is loaded through a separate
  backend context assembly step.
- Result ordering is deterministic for equal scores.

### Context Contract

Search finds candidates. Context assembly renders bounded source blocks.

The assistant tool loop uses an internal backend operation:

```text
fetch_retrieved_context(context_refs)
```

This operation is not a user-visible public API in the first cut. It renders
quote-safe XML-like context blocks using typed source refs and the current
viewer authorization.

Rendered context blocks must include:

- source type
- stable source ID
- media ID when present
- title/source label
- exact quote or transcript excerpt
- surrounding text when available
- timestamp or fragment locator when available
- deep link

### Conversation State

Assistant retrieval is distinct from user-attached context.

User-attached context remains explicit user intent. Assistant-retrieved context
is persisted as retrieval/tool metadata connected to the assistant message.

Persisted records include:

- conversation ID
- user message ID
- assistant message ID
- tool name
- tool call index
- query hash, not raw query text by default
- scope
- requested types
- semantic flag
- result refs
- selected context refs
- latency
- status
- provider request IDs when available

### Result Types

The final result taxonomy is:

- `media`: saved media, including videos and podcast episodes.
- `podcast`: show-level podcast result.
- `fragment`: document, article, ebook, PDF, video transcript, or podcast
  transcript fragment.
- `annotation`: user annotation with highlight context.
- `message`: visible conversation message.
- `transcript_chunk`: semantic transcript chunk for any transcript-bearing
  media, not podcast-only.

The hard cutover generalizes the current podcast-only semantic chunk path into
a media transcript chunk path. There is no retained podcast-only semantic
search implementation after cutover.

### Scopes

`all` searches all visible saved content.

`media:<id>` searches one readable media item and its related fragments,
annotations, and transcript chunks.

`library:<id>` searches media and derived content in a readable library.

`conversation:<id>` searches visible messages in the conversation and media
associated with that conversation. This replaces the old narrower
conversation-only-message behavior.

Unauthorized scopes return not found semantics and do not leak existence.

## Architecture

### Layers

Browser:

- Renders tool activity, citations, source chips, and deep links.
- Does not execute search business logic.

Next.js BFF:

- Proxies authenticated requests to FastAPI.
- Does not inspect, rewrite, rank, authorize, or normalize search results.

FastAPI routes:

- Validate request shape.
- Call services.
- Return response envelopes or SSE events.

Services:

- Own tool orchestration, search execution, context assembly, persistence, and
  authorization.

LLM adapters:

- Translate provider-neutral tool specs and tool results into provider-specific
  request/response formats.
- Do not know Nexus search semantics.

### Runtime Flow

Non-streaming:

1. Persist user message.
2. Build provider-neutral LLM request with available tools.
3. Model returns either tool calls or a final answer.
4. Backend validates tool call args.
5. Backend executes `app_search` with viewer ID and conversation ID.
6. Backend persists tool metadata.
7. Backend sends tool results back to the model.
8. Repeat within budget.
9. Backend persists final assistant answer and citations.
10. API returns conversation, messages, tool summaries, and citations.

Streaming:

1. Same orchestration semantics as non-streaming.
2. SSE emits structured events for tool calls and tool results.
3. Final text still streams as deltas.
4. Tool state and final answer persist even if the browser disconnects.

Transport does not own application work lifecycle. Streaming reconnects must not
duplicate completed tool work.

## Rules

- One primary search capability. Do not create a second search stack for the
  agent.
- Tool execution is read-only in this work.
- Every tool call is authorized with the current viewer ID.
- Visibility filtering happens inside database-backed search queries or
  service authorization checks before model exposure.
- The model never receives raw ORM objects or raw SQL result bags.
- The model never receives more context than the backend context budget allows.
- Store query hashes by default. Do not log raw user search queries.
- Tool calls have max count, timeout, result count, and context character
  budgets.
- Tool responses include only high-signal fields.
- Source citations are required for claims based on retrieved app content.
- Provider adapters implement the same provider-neutral tool contract.
- Models/providers that cannot satisfy required tool semantics are not offered
  as app-search-capable chat models.
- All old prompt-only assumptions in app-search-capable send paths are removed
  in the cutover.
- Existing tests are updated to the new behavior instead of preserving legacy
  behavior.

## Key Decisions

### Use First-Party App Search

Nexus app search remains first-party because app content is relational,
permissioned, typed, and deep-linkable. Hosted vector stores are not the
canonical source for ACL-sensitive app content.

### Use One Read-Only Tool First

The first cut exposes only `app_search`. More tools increase ambiguity and
agency risk. Follow-up tools require a separate spec.

### Separate Search From Context Assembly

Search returns candidates. Backend context assembly decides which source text is
safe and useful to place in the prompt. This keeps result pages fast and model
context bounded.

### Generalize Transcript Chunks

Semantic transcript search must support podcasts and videos through the same
transcript chunk abstraction. The current podcast-only path is removed during
cutover.

### Persist Retrievals Separately From Attachments

User-attached context and assistant-retrieved context represent different
intent. They must not share one overloaded schema.

### Prefer Deterministic Retrieval Before Agentic Loops

The first implementation can run one required search when intent is obvious.
The provider-native tool loop is still the final state, but the orchestrator
owns all budgets and state.

## Implementation Plan

### 1. Schema and Types

- Add provider-neutral tool dataclasses.
- Add `ToolCall`, `ToolResult`, and tool-stream event shapes.
- Add persisted tool/retrieval models and migrations.
- Add typed `RetrievedContextRef` separate from `MessageContextRef`.
- Add search result schemas for `podcast` and generalized
  `transcript_chunk`.

### 2. Retrieval Kernel

- Move model-facing result shaping into `agent_tools/app_search.py`.
- Reuse `services/search.py` for ranking and visibility.
- Add conversation scope behavior for conversation-associated media.
- Generalize semantic transcript chunk search from podcast-only to
  transcript-bearing media.
- Add canonical deep-link generation for every returned result type.

### 3. Context Assembly

- Add retrieved-context rendering for fragment, message, transcript chunk,
  podcast, and media results.
- Keep existing highlight/annotation rendering behavior but route through the
  new retrieved context path when the source came from search.
- Enforce context count and character budgets before LLM calls.

### 4. Tool Orchestration

- Add a backend tool loop shared by streaming and non-streaming send paths.
- Validate tool args with Pydantic before execution.
- Enforce max tool calls, max search calls, timeout, result count, and context
  limits.
- Persist tool calls and retrieval records.
- Return tool results to the model through provider-neutral abstractions.

### 5. Provider Adapters

- Implement tool request and tool response translation for each supported
  provider.
- Normalize provider tool-call events into one internal shape.
- Remove app-search-capable models from the selectable set if their provider
  adapter cannot support the required semantics.

### 6. Frontend

- Render tool activity and source citations in chat.
- Add link handling for every result type.
- Keep `/api/search` as the user search endpoint and BFF proxy.
- Reuse the canonical search row/link adapter where possible.
- Update streaming client handling for `tool_call` and `tool_result` events.

### 7. Tests and Cutover

- Write failing tests from the acceptance criteria first.
- Update existing chat/search tests to the new final behavior.
- Remove tests that assert legacy no-tool behavior for app-search-capable send.
- Run backend, frontend, and E2E verification gates.

## Acceptance Criteria

### Backend Search

- `app_search` returns media, podcast, fragment, annotation, message, and
  transcript chunk results with stable typed refs.
- Invisible media, annotations, messages, libraries, conversations, and
  transcript chunks never appear in results.
- Unauthorized scopes return not found semantics.
- `conversation:<id>` searches visible conversation messages and associated
  conversation media.
- Semantic transcript search works for podcast episodes and videos.
- Result links open the correct media, fragment, message, or timestamp target.
- Search logs include query hashes, not raw queries.

### Tool Orchestration

- The assistant calls `app_search` for saved-content questions.
- The assistant can make bounded follow-up searches.
- Tool calls are rejected when args are invalid, over budget, or unauthorized.
- Tool calls and retrieval records persist with status and timing.
- A failed tool call produces a controlled assistant response, not a crash or
  hallucinated citation.
- Non-streaming and streaming paths produce equivalent final persisted state.
- Browser disconnect during streaming does not duplicate completed tool work.

### Answer Quality

- Answers based on app content include citations.
- Citations point to exact Nexus sources.
- The assistant says it found no relevant saved content when retrieval returns
  no useful results.
- The assistant does not cite sources that were not returned by retrieval.
- The assistant does not claim searched-app evidence when no search ran.

### Frontend

- Chat shows search activity while the assistant searches.
- Chat shows cited source chips on final answers.
- Citation chips deep-link to media, timestamps, fragments, annotations, or
  conversation messages as applicable.
- Existing user search UI still works through the same canonical search API.

### Tests

- Backend integration tests cover visibility, scopes, result types, tool
  orchestration, persistence, and failure modes.
- Frontend tests cover tool event rendering and citation link behavior.
- E2E covers at least one saved media question and one no-results question.
- The final verification command for the PR passes.

## Cutover Rules

- Remove old prompt-only app-content answer behavior for app-search-capable
  models.
- Remove podcast-only semantic transcript search code after generalized
  transcript chunks land.
- Remove duplicate result-to-link mapping if a canonical mapper replaces it.
- Do not keep compatibility branches for old context refs in the send path.
- Do not keep old SSE event parsing branches once the new stream protocol lands.
- Do not ship feature flags for legacy behavior.

## Open Questions

- Which providers and models are app-search-capable on day one?
- Should raw query text ever be stored behind an admin-controlled debug mode, or
  should retrieval records remain hash-only permanently?
- Should `app_search` be automatically invoked for every user turn, or only
  when intent classification says saved app content is relevant?
- Should source chips expose the exact tool query to the user, a summarized
  query, or only result metadata?
